# P7 沙箱与安全：基础知识讲解

## 1. 这个技术是什么

沙箱 Sandbox 是给 Agent 工具执行加上的安全隔离层。

它限制 Agent 能做什么、能访问哪里、能运行多久、能不能联网、能不能读取敏感文件。

```text
Agent tool_call
  ↓
ToolRiskPolicy
  ↓
ApprovalPolicy
  ↓
SandboxPolicy
  ↓
SandboxExecutor
  ↓
受限执行环境
```

## 2. 为什么要用沙箱

漫剧 Agent 已经具备这些能力：

- `bash`
- `python_exec`
- `write_file`
- `edit_file`
- `http_request`
- 读写本地文件
- 访问外部 API

这些能力很强，但也危险。

没有沙箱时：

```text
LLM 误调用 bash
  ↓
删除文件 / 读取 .env / 外发密钥 / 无限循环
```

## 3. 好处

- **降低破坏风险**：限制文件写入范围。
- **保护密钥**：避免读取 `.env`、API Key。
- **控制网络**：禁止未知外连或只允许白名单。
- **控制资源**：限制 timeout、输出大小、子进程。
- **可审计**：记录命令、cwd、参数和结果。

## 4. 弊端

- **实现复杂**：不同工具需要不同策略。
- **可能影响灵活性**：过严会导致任务无法完成。
- **维护成本高**：白名单、路径策略、命令策略要持续更新。
- **真正强隔离需要容器**：纯 Python 限制不是强安全边界。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 只靠用户审批 | 简单 | 用户不一定懂风险 |
| 代码级 allowlist | 实现快 | 隔离不强 |
| 本地临时目录沙箱 | 适合文件限制 | 不能完全隔离进程 |
| Docker 沙箱 | 隔离强 | 部署复杂 |
| Firecracker/gVisor | 安全强 | 成本和复杂度高 |
| OpenHands 类 workspace | 工程化强 | 需要完整工具系统 |

## 6. 图示

```text
危险工具：bash/python/write/http
          ↓
  RiskPolicy 判断风险等级
          ↓
  ApprovalPolicy 判断是否审批
          ↓
  SandboxPolicy 注入限制
          ↓
  执行：timeout + cwd + env + fs allowlist + network allowlist
```

## 7. 关键点

审批不是沙箱。

```text
用户确认 ≠ 安全隔离
```

生产级 Agent 必须同时具备：

```text
审批 + 沙箱 + 审计 + 回滚策略
```
