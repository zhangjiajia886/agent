# P4 CompletionAuditor：基础知识讲解

## 1. 这个技术是什么

CompletionAuditor 是任务完成审计器。

它负责判断：

```text
任务到底完成了吗？
失败了吗？
阻塞了吗？
还有哪些步骤没做？
```

## 2. 为什么要用 Auditor

旧方式依赖模型文本：

```text
模型说“任务完成” → 系统相信完成
```

这很危险，因为模型可能：

- 工具失败后仍说完成。
- 只列计划没执行也说完成。
- 忘记用户要求的视频/音频步骤。

Auditor 用结构化状态判断：

```text
required steps 全部 succeeded
required artifacts 全部 verified
才 completed
```

## 3. 好处

- **防误判完成**。
- **能列出剩余步骤**。
- **能区分 failed / blocked / incomplete**。
- **让 FinalReport 更可信**。

## 4. 弊端

- **规则要维护**。
- **主观质量难判断**：比如图片是否“好看”。
- **依赖 Planner 准确性**：required step 标错会影响审计。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| LLM 自评 | 解释能力强 | 不可靠 |
| 前端文本扫描 | 简单 | 极易误判 |
| 规则 Auditor | 稳定可信 | 需要维护规则 |
| LLM Critic | 可评估质量 | 成本高、不稳定 |
| 人工验收 | 最可靠 | 不自动化 |

## 6. 图示

```text
TaskGraph + Steps + Artifacts
          ↓
   CompletionAuditor
          ↓
completed / incomplete / blocked / failed / canceled
          ↓
      Reporter 生成人话报告
```

## 7. 关键点

LLM 可以负责解释，但不能作为唯一判定来源。

```text
规则决定状态，LLM 解释状态
```
