# P6 恢复与观测：基础知识讲解

## 1. 这个技术是什么

恢复与观测包含三类能力：

- **断线恢复**：WebSocket 断了还能恢复任务状态。
- **事件回放**：用历史事件重建前端工作台。
- **Tracing**：记录每一步执行轨迹，便于排查。

## 2. 为什么要用恢复与观测

漫剧任务通常耗时长：

- 图片生成可能几十秒。
- 视频生成可能几分钟。
- TTS/合成也可能失败。

如果没有恢复：

```text
页面刷新 → 任务过程丢失
WebSocket 断开 → 用户不知道结果
工具失败 → 无法复盘
```

## 3. 好处

- **用户体验更稳定**。
- **长任务可恢复**。
- **失败可排查**。
- **可统计成功率和耗时**。
- **为后续自动优化提供数据**。

## 4. 弊端

- **事件数据量大**。
- **前端回放逻辑复杂**。
- **需要任务控制 API**。
- **隐私和日志脱敏要注意**。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 不恢复 | 简单 | 用户体验差 |
| 只存最终结果 | 数据少 | 不能复盘过程 |
| 事件回放 | 可完整恢复 | 实现复杂 |
| LangSmith | 观测强 | 外部依赖/成本 |
| 自研 Tracing | 贴合业务 | 需要维护 |

## 6. 图示

```text
AgentEvent 表
  ↓
按时间排序读取
  ↓
前端逐条 replay
  ↓
重建：steps + artifacts + logs + final_report
```

## 7. 关键点

事件回放不是重新执行任务，而是重放历史状态变化。

```text
Replay events ≠ Re-run tools
```
