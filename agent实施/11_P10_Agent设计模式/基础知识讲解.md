# P10 Agent 设计模式：基础知识讲解

## 1. 这个技术是什么

Agent 设计模式是构建 Agent 系统时常用的架构套路。

它回答：

```text
Agent 怎么思考？
怎么规划？
怎么调用工具？
怎么保存状态？
怎么判断完成？
失败后怎么恢复？
怎么保证安全？
```

Agent 不是单一技术，而是多个模式组合。

## 2. 为什么要学习设计模式

如果只把 Agent 理解成：

```text
LLM + Tools
```

很快会遇到问题：

- 只说计划不执行。
- 工具失败后乱重试。
- 任务完成误判。
- 图片/视频路径丢失。
- 无法恢复任务。
- 自动执行有安全风险。

设计模式的作用是把这些问题拆开，用成熟结构解决。

## 3. 常见模式图谱

```text
ReAct
Tool Calling
Plan-and-Execute
TaskGraph / DAG
State Machine
Artifact-Centric
Event-Driven
Auditor
Replanner
Sandbox
Budgeted Agent
Memory-Augmented
Observer / Verifier
Multi-Agent
Critic / Reflection
```

## 4. 好处

- **降低复杂度**：每个模式解决一类问题。
- **方便沟通**：团队知道当前采用什么架构。
- **便于演进**：可以分阶段引入。
- **减少返工**：避免用 prompt 修补架构问题。

## 5. 弊端

- **模式过多会复杂化**。
- **不适合过早全量引入**。
- **需要结合业务裁剪**。
- **模式之间要明确边界**。

## 6. 竞争方案 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 纯 Prompt Agent | 快速 | 不可靠 |
| ReAct 单模式 | 简单 | 长任务弱 |
| LangChain Agent | 生态成熟 | 状态和产物仍需自研 |
| LangGraph | 状态图强 | 接入成本和抽象成本高 |
| 自研组合模式 | 最贴合业务 | 工程纪律要求高 |

## 7. 当前推荐组合

对 ttsapp 漫剧 Agent，推荐组合是：

```text
ReAct + Tool Calling + TaskGraph + State Machine
+ Artifact-Centric + Event-Driven
+ Auditor + Replanner
+ Sandbox + Budget + DB/Redis 状态层
```

## 8. 关键点

不要追求“最复杂 Agent”，而是构建“最可靠的领域 Agent”。

```text
模式为业务服务，不为炫技服务。
```
