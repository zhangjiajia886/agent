# P8 工具治理：基础知识讲解

## 1. 这个技术是什么

工具治理 Tool Governance 是对 Agent 可用工具进行注册、描述、权限、风险、能力、成本、健康状态和版本管理。

它回答：

```text
这个工具能做什么？
需要什么输入？
会产出什么？
谁可以调用？
是否要审批？
失败后用谁替代？
成本是多少？
能否并行？
```

## 2. 为什么要用工具治理

Agent 的能力来自工具。

没有治理时：

```text
LLM 看到一堆工具描述
  ↓
自己猜哪个工具合适
  ↓
参数填错 / 工具不可用 / 风险失控
```

有治理后：

```text
ToolRegistry + Capability Metadata
  ↓
Planner 生成正确 TaskGraph
  ↓
Scheduler 判断风险和并行
  ↓
Replanner 使用 fallback
```

## 3. 好处

- **工具可发现**：Planner 知道有哪些工具。
- **工具可组合**：根据输入/输出产物连接步骤。
- **工具可控**：风险等级和审批策略明确。
- **工具可恢复**：失败时有 fallback。
- **工具可观测**：统计成功率、耗时、成本。

## 4. 弊端

- **元数据维护成本高**。
- **工具升级需要同步能力描述**。
- **描述错误会影响 Planner/Scheduler**。
- **过度治理会降低灵活性**。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 纯 OpenAI tools schema | 标准、简单 | 缺少风险/成本/产物关系 |
| LangChain Tools | 生态丰富 | 治理能力要自研 |
| 手写工具列表 | 快 | 不可维护 |
| Tool Registry DB | 可管理、可查询 | 需要维护元数据 |
| MCP Tool Registry | 标准化潜力大 | 需要生态接入 |

## 6. 图示

```text
ToolRegistry
  ├── schema
  ├── risk_level
  ├── input_artifact_types
  ├── output_artifact_types
  ├── fallback_tools
  ├── cost_level
  └── health_status
          ↓
Planner / Scheduler / Replanner / Auditor
```

## 7. 关键点

工具治理不是工具执行本身，而是工具执行前后的“管理层”。

```text
工具执行解决能不能做
工具治理解决该不该做、怎么安全地做、失败后怎么办
```
