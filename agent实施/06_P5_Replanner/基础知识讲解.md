# P5 Replanner 与 RecoveryPolicy：基础知识讲解

## 1. 这个技术是什么

Replanner 是失败恢复模块。

当工具失败、输入缺失、用户拒绝或服务不可用时，它决定下一步：

```text
retry / fallback_tool / ask_user / skip / block / fail
```

## 2. 为什么要用 Replanner

没有 Replanner 时，失败恢复靠 LLM 自己猜：

```text
工具失败 → LLM 可能继续重试 → 可能无限循环
```

Replanner 用策略控制：

```text
失败可重试？
  是 → retry
  否 → 有 fallback？
          是 → fallback_tool
          否 → ask_user/block/fail
```

## 3. 好处

- **避免无限重试**。
- **自动切换 fallback 工具**。
- **缺输入时询问用户**。
- **失败路径可解释、可复盘**。

## 4. 弊端

- **策略表需要维护**。
- **fallback 工具能力可能不完全等价**。
- **自动恢复可能增加成本**。
- **需要和审批策略联动**。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| LLM 自我修复 | 灵活 | 不可控 |
| 固定 retry | 简单 | 不能处理根因 |
| 手动用户处理 | 安全 | 自动化差 |
| Replanner 策略 | 可控、可恢复 | 规则复杂 |
| 多 Agent 讨论 | 可能更聪明 | 成本高、状态难控 |

## 6. 图示

```text
ToolResult error
  ↓
retryable?
  ├── yes → retry_count < max → retry
  └── no / retry exhausted
        ↓
     fallback_tools?
        ├── yes → fallback_tool
        └── no → ask_user / block / fail
```

## 7. 关键点

必须有防循环预算：

- 每个 step 最大 retry。
- 每个工具最大调用次数。
- 用户拒绝后不重复调用同一工具。
- 总任务耗时预算。
