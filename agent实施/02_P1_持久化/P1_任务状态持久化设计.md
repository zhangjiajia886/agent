# P1 任务状态持久化设计

> 所属阶段：P1  
> 状态：最小实现已落地  
> 目标：把 P0 内存态 RuntimeTask 升级为数据库权威状态。

---

## 1. 背景

当前 P0 的 `RuntimeTask / RuntimeStep` 只存在于 `agent_stream` 内存中。页面刷新、WebSocket 断开、服务重启后，任务状态无法恢复。

P1 要将任务、步骤、产物、事件、工具调用全部落库。

## 6. 数据流

```text
agent_stream
  ↓
create_runtime_task
  ↓
store.create_task
  ↓
store.create_steps
  ↓
emit(event) → store.append_event → websocket.send_json
  ↓
tool_start → create_tool_invocation
  ↓
tool_done → finish_tool_invocation + create_artifact
  ↓
done → AgentTask.final_report
```

## 7. API 设计

```text
GET /api/v1/comic-agent/tasks/{task_uid}
GET /api/v1/comic-agent/tasks/{task_uid}/events
GET /api/v1/comic-agent/tasks/{task_uid}/artifacts
GET /api/v1/comic-agent/conversations/{conversation_id}/tasks
```

所有任务查询 API 必须基于 `current_user.id` 校验任务归属，避免跨用户读取任务、事件、产物和工具调用记录。

## 8. 风险

- WebSocket 中复用同一个 AsyncSession，长任务可能导致事务边界不清。
- 事件落库过多可能造成数据库写入压力。
- `delta/thinking` 事件如果全部落库会很大。

## 9. 策略

- P1 只落库结构化事件，不落库每个 `delta`。
- `thinking` 只落关键摘要。
- 工具调用必须落库。
- 产物必须落库。

## 10. 事务边界设计

P1 的关键原则是：**不要让整个 WebSocket 生命周期持有一个长事务**。

### 10.1 Store 短事务原则

所有 Store 写操作采用独立短事务：

```text
create_task            → 独立事务，立即 commit
create_steps           → 独立事务，立即 commit
append_event           → 独立事务，立即 commit
update_step_status     → 独立事务，立即 commit
create_tool_invocation → 独立事务，立即 commit
finish_tool_invocation → 独立事务，立即 commit
create_artifact        → 独立事务，立即 commit
finish_task            → 独立事务，立即 commit
```

### 10.2 推荐实现方式

Store 不复用 `agent_stream` 长生命周期 `db` 事务，而是内部使用 `AsyncSessionLocal` 创建短 session：

```python
class AgentTaskStore:
    async def append_event(self, event: dict) -> AgentEvent:
        async with AsyncSessionLocal() as db:
            row = AgentEvent(...)
            db.add(row)
            await db.commit()
            await db.refresh(row)
            return row
```

### 10.3 工具调用事务

工具调用拆成两段：

```text
tool_start
  ↓
create_tool_invocation(status='running') commit
  ↓
execute_tool(...)
  ↓
finish_tool_invocation(status='success/error') commit
```

这样即使工具执行中 WebSocket 断开，也能看到工具已开始。

### 10.4 事件落库事务

事件发送前先落库：

```text
build event
  ↓
store.append_event(event)
  ↓
websocket.send_json(event)
```

如果 WebSocket 发送失败，事件仍然可用于恢复和排查。

### 10.5 不落库或降采样事件

| 事件 | P1 策略 |
|---|---|
| `delta` | 不落库 |
| `thinking` | 只保存关键摘要，限制长度 |
| `task_created` | 必须落库 |
| `task_update` | 必须落库 |
| `step_update` | 必须落库 |
| `tool_start` | 必须落库 |
| `tool_done` | 必须落库 |
| `artifact_created` | 必须落库 |
| `done` | 必须落库 |

### 10.6 异常一致性策略

- 工具执行异常：必须写入 `ToolInvocation.error` 和 `AgentStep.error`。
- AgentRunner 异常：必须尝试写入 `AgentTask.status=failed`。
- WebSocket 断开：不立即把任务标记 failed，只记录连接断开事件，后续 P6 再处理恢复。
- 用户取消：显式写入 `canceled`。

## 11. 回滚方案

- 如果 Store 接入导致 WebSocket 异常，可临时关闭落库调用，保留 P0 内存态事件。
- 新增表不删除，保留数据用于排查。
- API 查询失败不影响原聊天主链路。
