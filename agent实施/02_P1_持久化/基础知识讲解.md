# P1 状态持久化：基础知识讲解

## 1. 这个技术是什么

状态持久化是把 Agent 运行时状态写入数据库。

包括：

- `AgentTask`：一次用户目标任务。
- `AgentStep`：任务拆解后的步骤。
- `AgentArtifact`：图片、视频、音频、文件等产物。
- `AgentEvent`：WebSocket 事件流。
- `ToolInvocation`：每次工具调用。

## 2. 为什么要用持久化

内存状态的问题：

```text
WebSocket 断开 → 状态丢失
页面刷新 → 前端丢失任务过程
后端重启 → 任务无法恢复
失败排查 → 没有历史记录
```

持久化后的能力：

```text
任务可查询
事件可回放
产物可追踪
失败可复盘
```

## 3. 好处

- **可恢复**：刷新页面后能重建任务视图。
- **可审计**：知道哪个工具失败。
- **可统计**：计算成功率、失败率、耗时。
- **可扩展**：为 Scheduler、Auditor、Replanner 提供数据源。

## 4. 弊端

- **写入复杂**：每个状态变化都要落库。
- **事务风险**：长 WebSocket 不能持有长事务。
- **数据量增加**：事件流可能很多。
- **迁移成本**：后续需要 Alembic 管理表结构。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 | 适用性 |
|---|---|---|---|
| 纯内存 | 简单、快 | 不可恢复 | 只适合 demo |
| Redis 状态 | 快、适合临时状态 | 历史查询弱 | 可做运行中状态 |
| MySQL 持久化 | 稳定、可查询 | 写入成本高 | 当前推荐 |
| Event Sourcing | 可完整回放 | 设计复杂 | 后续增强 |
| LangGraph Checkpoint | 框架内建 | 业务查询仍需自研 | 可参考 |

## 6. 图示

```text
AgentRunner
  ├── task_created ─────→ agent_task
  ├── step_update ──────→ agent_step
  ├── tool_start/done ──→ tool_invocation
  ├── artifact_created → agent_artifact
  └── all events ───────→ agent_event
```

## 7. 关键原则

不要让 WebSocket 生命周期持有一个长事务。

正确方式：

```text
每次 Store 写入 = 独立短事务 + 立即 commit
```
