# P9 数据库与 Redis 状态层：基础知识讲解

## 1. 这个技术是什么

数据库与 Redis 状态层是 Agent Runtime 的“状态基础设施”。

它负责区分：

```text
什么状态需要长期保存？
什么状态只是运行中临时协调？
什么状态丢了还能恢复？
什么状态必须绝对可靠？
```

在本系统里：

```text
数据库 = 权威长期状态
Redis = 临时运行状态 / 缓存 / 锁 / 计数器 / 队列
```

## 2. 为什么要同时用数据库和 Redis

只用数据库的问题：

- 高频计数和锁性能一般。
- WebSocket 在线状态不适合长期表。
- 审批等待、任务锁这种临时状态写 DB 太重。

只用 Redis 的问题：

- Redis 数据可能过期或丢失。
- 不适合作为最终审计记录。
- 历史任务查询、分页、统计不方便。

所以推荐组合：

```text
DB 保存真相
Redis 加速运行
```

## 3. 数据库应该保存什么

数据库保存长期、可审计、可恢复的数据：

- `AgentTask`
- `AgentStep`
- `AgentArtifact`
- `AgentEvent`
- `ToolInvocation`
- `FinalReport`
- `ToolRegistry`

图示：

```text
AgentTask
  ├── AgentStep
  ├── AgentArtifact
  ├── AgentEvent
  └── ToolInvocation
```

## 4. Redis 应该保存什么

Redis 保存短期、高频、可过期数据：

- WebSocket 连接状态
- running task lock
- approval 等待状态
- budget 临时计数
- tool health cache
- rate limit
- cancel/resume 信号
- 简单任务队列

图示：

```text
Redis
  ├── agent:task:{task_uid}:lock
  ├── agent:task:{task_uid}:approval
  ├── agent:task:{task_uid}:budget
  ├── agent:tool:{tool_name}:health
  └── agent:rate:user:{user_id}:minute
```

## 5. 好处

- **可恢复**：DB 中有完整任务历史。
- **高性能**：Redis 处理锁、限流、临时状态。
- **职责清晰**：DB 是真相，Redis 是加速。
- **支撑长任务**：断线恢复、审批等待、任务取消更好做。

## 6. 弊端

- **一致性更复杂**：DB 和 Redis 要明确主从关系。
- **运维多一个组件**：Redis 需要启动、监控、配置。
- **缓存过期策略要设计**。
- **如果误把权威状态放 Redis，会丢数据**。

## 7. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 纯数据库 | 简单、可靠 | 锁/限流/临时状态不灵活 |
| 纯 Redis | 快 | 不适合长期审计 |
| DB + Redis | 可靠 + 高性能 | 一致性复杂 |
| 消息队列 | 适合异步任务 | 不能替代状态存储 |
| LangGraph checkpoint | Runtime 状态强 | 业务查询仍需 DB |

## 8. 关键原则

```text
DB 是权威状态。
Redis 是临时协调状态。
Redis 丢失不能导致任务历史丢失。
```
