# F3 审批交互：基础知识讲解

## 1. 这个技术是什么

审批交互（Human-in-the-Loop Approval）是 Agent 在执行高风险操作前暂停，请求用户确认后再继续的交互模式。类似 CI/CD 的手动审批门或 sudo 确认。

## 2. 为什么要用

- Agent 可能执行费钱的操作（GPU 渲染、API 调用）。
- 删除或修改文件等不可逆操作需要确认。
- 参数错误时提前拦截，避免浪费资源。

## 3. 好处

- **安全**：高风险操作有人类把关。
- **可控**：用户对 Agent 行为有最终决策权。
- **纠错**：发现参数错误时可以 reject 并提供修正建议。

## 4. 弊端

- **中断体验**：自动模式下频繁审批很烦。
- **超时风险**：用户不及时响应会阻塞任务。
- **决策负担**：用户不一定理解技术参数。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 全自动（无审批）| 快 | 不安全 |
| 全审批 | 安全 | 体验差 |
| 基于风险等级审批 | 平衡 | 需要准确的风险评估 |
| 白名单免审 | 灵活 | 需要维护白名单 |
| auto_mode 开关 | 用户选择 | 用户可能不了解风险 |

## 6. 图示

```text
Agent 决定调用工具
  ↓
SandboxChecker.check()
  ↓
risk_level >= 2 && !auto_mode?
  ├── YES → 暂停 → 发送 tool_confirm → 等待用户
  └── NO  → 直接执行
```

## 7. 关键点

审批交互的目的不是阻止 Agent 工作，而是在关键节点给用户知情权和决策权。

```text
Approval = Safety Gate, not Blocker
```
