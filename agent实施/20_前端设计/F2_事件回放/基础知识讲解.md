# F2 事件回放：基础知识讲解

## 1. 这个技术是什么

事件回放（Event Replay）是指从持久化的事件日志中读取历史事件，重新驱动 UI 状态，使前端恢复到断连前的最新状态。核心思想来自 Event Sourcing。

## 2. 为什么要用

- WebSocket 断连后用户刷新页面，任务可能还在后台运行。
- 不回放就看不到之前的步骤和产物。
- 用户体验要求「刷新不丢状态」。

## 3. 好处

- **无损恢复**：断连后完整恢复任务面板。
- **调试友好**：可以回看任务执行历史。
- **架构简洁**：前端只需要一个 `dispatch(event)` 入口。

## 4. 弊端

- **事件量大时回放慢**：长任务可能有上百条事件。
- **状态一致性**：回放顺序必须正确。
- **回放与实时交叉**：回放过程中可能收到新实时事件。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 不回放，只显示最终态 | 简单 | 丢失过程 |
| 轮询 REST API | 不需要 WebSocket | 延迟高 |
| Event Replay | 完整恢复 | 实现复杂 |
| Snapshot + Delta | 快速恢复 | 需要维护快照 |

## 6. 图示

```text
历史事件 DB
  ↓ GET /events
[task_created, step_update, tool_done, artifact_created, ...]
  ↓ replay()
TaskWorkspace UI 恢复
  ↓ WebSocket 重连
继续接收实时事件
```

## 7. 关键点

回放 ≠ 重新执行。回放只是把历史状态变化重新 dispatch 到 UI store。

```text
Replay events ≠ Re-run tools
```
