# F6 Trace 时间线：基础知识讲解

## 1. 这个技术是什么

Trace 时间线是将 Agent 执行过程中的所有事件按时间顺序可视化的调试和分析工具。类似 Chrome DevTools 的 Performance Timeline 或 Jaeger/Zipkin 的分布式追踪视图。

## 2. 为什么要用

- 开发者和运维需要排查任务失败原因。
- 用户想了解 Agent 做了什么、花了多久。
- 性能分析：找出耗时最长的步骤。

## 3. 好处

- **全链路可见**：LLM 调用 → 工具执行 → 步骤状态 → 审计全部可追踪。
- **排查方便**：一眼看出哪步失败、为什么失败。
- **性能优化**：发现瓶颈工具。

## 4. 弊端

- **数据量大**：长任务可能有很多 trace 条目。
- **UI 复杂**：不同类型的事件需要不同展示方式。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 纯文本 log | 简单 | 不直观 |
| 时间线视图 | 直观 | 实现复杂 |
| 火焰图 | 适合性能分析 | 不适合状态追踪 |
| Jaeger/LangSmith | 功能强 | 外部依赖 |

## 6. 图示

```text
GET /tasks/{uid}/trace
  ↓
tool_invocations + events
  ↓
按时间排序合并
  ↓
TraceTimeline 组件
  ↓
TraceItem × N（不同类型不同图标）
```

## 7. 关键点

Trace 时间线是事后分析工具，不影响实时执行。

```text
Trace = Post-mortem analysis, not real-time control
```
