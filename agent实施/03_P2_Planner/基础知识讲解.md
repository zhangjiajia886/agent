# P2 TaskPlanner：基础知识讲解

## 1. 这个技术是什么

TaskPlanner 是把用户自然语言目标转换为结构化任务图的模块。

例如用户说：

```text
生成一张仙侠图，转视频，并配旁白
```

Planner 输出：

```text
s1 generate_image
s2 image_to_video depends_on s1
s3 text_to_speech
s4 merge_media depends_on s2,s3
```

## 2. 为什么要用 Planner

没有 Planner 时，LLM 自己记住下一步：

```text
messages 里记计划 → LLM 自己决定下一步
```

问题是：

- 容易忘记步骤。
- 容易只说计划不执行。
- 不知道哪些步骤可以并行。
- 工具失败后不知道应该重试还是换工具。

Planner 的目标是：

```text
让系统显式知道应该做什么
```

## 3. 好处

- **任务结构化**：自然语言变成 TaskGraph。
- **依赖明确**：视频依赖图片，合成依赖视频和音频。
- **可调度**：Scheduler 可以执行 ready step。
- **可审计**：Auditor 能判断 required step 是否完成。

## 4. 弊端

- **计划可能僵化**：用户需求变化时要重规划。
- **模板维护成本**：不同任务需要不同模板。
- **工具能力依赖**：必须知道工具输入/输出类型。
- **LLM 规划不稳定**：不能完全依赖 LLM 自由规划。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 纯 ReAct | 灵活 | 状态不可靠 | 不适合复杂任务 |
| 规则 Planner | 稳定、可控 | 覆盖有限 | 首期推荐 |
| LLM Planner | 泛化强 | 不稳定 | 只做补全 |
| LangGraph StateGraph | 状态机强 | 框架成本 | 可参考 |
| Workflow DAG 编辑器 | 可视化强 | 不适合即时对话 | 可融合 |

## 6. 图示

```text
用户目标
  ↓
规则模板
  ↓
工具能力校验
  ↓
LLM 补全参数
  ↓
TaskGraph
  ↓
Scheduler
```

## 7. 关键点

Planner 必须读取工具能力元数据：

```json
{
  "input_artifact_types": ["image"],
  "output_artifact_types": ["video"],
  "fallback_tools": ["jimeng_generate_video"]
}
```

否则 Planner 不知道工具之间如何连接。
