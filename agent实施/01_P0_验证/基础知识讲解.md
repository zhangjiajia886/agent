# P0.5 状态闭环验证：基础知识讲解

## 1. 这个技术是什么

P0.5 验证不是新功能，而是验证当前 P0 的“状态闭环”是否真实可用。

状态闭环指：

```text
后端创建任务
  ↓
后端更新步骤
  ↓
后端执行工具
  ↓
后端登记产物
  ↓
后端生成最终报告
  ↓
前端展示权威状态
```

## 2. 为什么要用状态闭环

旧模式中，前端靠文本猜测任务状态：

```text
模型说“完成了” → 前端认为完成
```

这是不可靠的。

新模式要求：

```text
数据库/后端状态显示完成 → 才是真的完成
```

## 3. 好处

- **减少误判**：模型只列计划不执行时，不会显示完成。
- **可观测**：每个步骤都有状态。
- **可扩展**：后续能接持久化、调度、恢复。
- **前端更简单**：前端只消费事件，不负责判断业务完成度。

## 4. 弊端

- **事件更多**：需要处理 `task_created/step_update/artifact_created`。
- **状态同步复杂**：前后端要保持协议一致。
- **短期有双逻辑**：旧事件兼容逻辑仍存在。

## 5. 竞争技术 / 替代方案

| 方案 | 说明 | 优点 | 缺点 |
|---|---|---|---|
| 文本推断 | 前端读 assistant 文本判断状态 | 简单 | 极易误判 |
| ReAct 自我判断 | LLM 自己说是否完成 | 灵活 | 不稳定 |
| 后端状态闭环 | 后端事件驱动任务状态 | 可靠 | 实现复杂 |
| LangGraph 状态机 | 用框架表达状态流 | 成熟 | 引入成本高 |

## 6. 图示

```text
旧模式：
LLM 文本 ──→ 前端猜测 ──→ completed?

新模式：
ToolResult ──→ Step 状态 ──→ Task 状态 ──→ FinalReport
```

## 7. 什么时候算验证通过

- 收到 `task_created`。
- step 能从 `running` 到 `succeeded/failed`。
- 产物通过 `artifact_created` 登记。
- `done.final_report.status` 与实际执行一致。
