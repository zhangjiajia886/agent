# P3 Scheduler 与 StepExecutor：基础知识讲解

## 1. 这个技术是什么

Scheduler 是任务调度器，决定哪个 step 现在可以执行。

StepExecutor 是步骤执行器，只负责执行当前 step。

```text
TaskGraph
  ↓
Scheduler 找 ready step
  ↓
StepExecutor 执行当前 step
  ↓
ToolExecutor 调工具
```

## 2. 为什么要用 Scheduler

如果没有 Scheduler，LLM 会自己决定下一步：

```text
LLM：我觉得下一步应该转视频
```

但如果图片还没生成，转视频必然失败。

Scheduler 用依赖关系判断：

```text
s2 depends_on s1
只有 s1 succeeded，s2 才能 ready
```

## 3. 好处

- **防跳步**：依赖没满足不执行。
- **支持并行**：无依赖的步骤可并行。
- **更稳定**：LLM 不再控制全局流程。
- **便于恢复**：从 DB step 状态即可恢复执行位置。

## 4. 弊端

- **实现复杂**：要维护状态机。
- **计划错误会影响执行**：Planner 生成错依赖，Scheduler 也会错。
- **并行有资源风险**：GPU/API 并发需要限制。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| LLM 自由 ReAct | 灵活 | 容易跳步、空转 |
| 顺序 Workflow | 简单 | 不支持并行和复杂依赖 |
| DAG Scheduler | 稳定、可并行 | 实现复杂 |
| LangGraph | 状态机成熟 | 接入成本高 |
| Celery/任务队列 | 分布式强 | 对 Agent 状态表达不直接 |

## 6. 图示

```text
s1 generate_image ──┐
                    ├── s3 merge_media
s2 text_to_speech ──┘

ready 初始：s1, s2
s3 等待：s1+s2 succeeded
```

## 7. 关键点

Scheduler 决定“做哪一步”，LLM 只决定“这一步的参数怎么填”。

```text
系统控流程，模型控局部执行
```
