# Agent 改造设计

## 一、需求分析

### 1.1 当前核心问题

结合你给出的界面截图、当前项目代码结构，以及现有 `ComicAgentView.vue` / `agent_runner.py` 的事件机制，当前 Agent 的主要问题不是单点样式不好看，而是**整套信息架构和视觉优先级都错位了**。

当前问题集中在以下几点：

- **信息顺序错误**
  - 用户最关心的是：
    - 我的问题被怎么理解了
    - 任务将如何推进
    - 当前做到哪一步了
    - 最终产出了什么
  - 但现在界面里，大量 `thinking`、工具卡片、结果卡片被混合平铺在消息流中，导致**高价值信息没有被放到最上层**。

- **“需求分析”没有成为第一视觉焦点**
  - 你明确提出：**第一件事应该是需求分析**。
  - 现在虽然有“已识别需求 / 已规划任务”之类的块，但它们只是普通聊天消息的一部分，没有形成一个稳定的任务头部区域，也没有成为整个执行过程的主轴。

- **思考与执行位置关系反了**
  - 你明确指出：**不要思考在下面，改动在上面，很奇怪**。
  - 当前 UI 中，`thinking` 作为一类普通消息插入聊天流，工具执行卡和结果卡又会不断插队，用户看到的是“细碎过程”，而不是“任务主线”。
  - 这会造成一种典型问题：**执行痕迹很多，但任务结构不清楚**。

- **视觉层级失控**
  - 当前消息大量使用相似卡片样式，`thinking / tool_start / tool_done / assistant` 都在同一纵向流里竞争注意力。
  - 结果是：
    - 重要节点不够突出
    - 次要过程太抢眼
    - 用户需要自己理解“哪些该看，哪些可以跳过”

- **不像成熟品牌 Agent，更像调试日志面板**
  - 从产品感受上说，当前界面更接近“开发过程事件流”而不是“任务执行界面”。
  - 成熟 Agent 的共性是：
    - 先给任务理解
    - 再给执行路径
    - 再给阶段进度
    - 最后才暴露可展开的细节
  - 而不是把所有中间事件一股脑堆在聊天区。

### 1.2 本次改造目标

本次改造不是简单换皮，而是要把漫剧 Agent 从“消息流 UI”升级为“任务执行 UI”。

目标如下：

- **目标一：建立自上而下的信息主轴**
  - 顶部先显示：需求分析、任务计划、当前阶段、最终产出
  - 详细思考、工具参数、执行日志放入次级区域

- **目标二：让用户第一眼看懂 Agent 在干什么**
  - 不要求用户阅读大段 `thinking`
  - 不要求用户自己拼接工具执行顺序
  - 关键步骤必须结构化、摘要化、状态化

- **目标三：对标知名品牌 Agent 的任务型体验**
  - 学习 ChatGPT Agent、Perplexity Labs、Devin 这类产品的共同点：
    - 任务导向
    - 进度可见
    - 工具使用可见但不喧宾夺主
    - 可随时接管
    - 结果与过程分层展示

- **目标四：适配漫剧/生图/编辑/多步骤链式任务**
  - 不能只适合问答
  - 必须适配当前系统真实场景：
    - 文生图
    - 图像编辑
    - 图生视频
    - 多工具串行
    - 需要审批的任务
    - 多媒体产物回流

### 1.3 用户真实诉求翻译

把你的要求翻译成产品设计语言，就是：

- **从聊天记录视角，改为任务面板视角**
- **从“底层过程堆叠”，改为“高层结论优先”**
- **从“思考暴露过多”，改为“摘要先行、细节折叠”**
- **从“像日志”，改为“像品牌 Agent”**

这将直接决定后续的版式结构：

1. 顶部固定任务头部
2. 任务主线卡片置顶
3. 当前阶段与状态可见
4. 结果产物集中展示
5. 细节日志折叠收纳到底部

## 二、对标知名品牌 Agent 的结论

### 2.1 ChatGPT Agent 的可借鉴点

依据公开介绍与产品摘要，ChatGPT Agent 的核心体验特点是：

- **协作式任务推进**
  - Agent 在执行时，用户可以随时打断、补充、修正目标
- **任务过程可见，但不过度暴露内部噪音**
  - 用户看到的是关键进展更新，而不是海量底层推理
- **任务执行是连续主线，而不是离散消息**
  - 更像“一个任务正在被完成”，而不是“很多消息正在产生”

对本项目的启发：

- 漫剧 Agent 必须把“本轮任务”作为一个明确对象来展示
- 需求分析、执行计划、当前阶段应该始终挂在任务主线上
- 允许用户中途修正，但不能破坏主线结构

### 2.2 Perplexity Labs 的可借鉴点

从产品公开介绍与摘要可以提炼出一个很重要的原则：

- **任务、来源、产物、步骤是分区呈现的**
- 用户不会在一个单一聊天流里同时承受所有信息
- 复杂任务被拆解成几个清晰区域：
  - Tasks
  - Sources
  - Assets
  - Images / 输出物

对本项目的启发：

- 漫剧 Agent 不应该把：
  - 需求分析
  - 执行计划
  - 工具确认
  - 过程日志
  - 图片/视频结果
  全都塞进同一条消息流
- 应该拆成明显的功能区域

### 2.3 Devin 的可借鉴点

从 Devin 文档中能确认两个非常关键的点：

- **Progress Tab**
  - 进度本身是一个独立视图，而不是普通对话文本
- **Typical workflow**
  - 典型工作流是：
    1. 开始任务
    2. 监控 progress updates
    3. 查看 shell commands
    4. 在 IDE/diff 中看改动
    5. 需要时接管
    6. 再继续协作

对本项目的启发：

- “进度”必须成为独立模块，不应只是 `thinking` 文本
- “工具调用”和“改动细节”应从主消息流中降级，变成可展开的执行细节
- 用户应该始终知道自己当前在看的是：
  - 任务摘要
  - 当前阶段
  - 过程细节
  - 最终结果

### 2.4 对标后的统一设计原则

综合以上品牌 Agent，可以提炼出本项目必须遵守的 6 条原则：

1. **需求分析优先于过程细节**
2. **任务主线优先于消息时间线**
3. **阶段状态优先于底层日志**
4. **结果展示优先于过程噪音**
5. **可展开细节，而不是默认灌输细节**
6. **一个任务，一个主视图，而不是几十条离散消息**

## 三、当前系统的结构现状

### 3.1 前端现状

当前 `frontend/src/views/comic-agent/ComicAgentView.vue` 的本质，是一个**事件型消息流渲染器**。

核心现状：

- `AgentEvent` 类型包括：
  - `thinking`
  - `delta`
  - `text`
  - `tool_confirm`
  - `tool_start`
  - `tool_done`
  - `error`
  - `done`
- `handleAgentEvent()` 会把这些事件直接转成 `messages[]`
- 最终 UI 用消息流顺序逐条渲染

这意味着：

- 当前 UI 的组织方式是“事件驱动消息追加”
- 而不是“任务对象驱动的分区渲染”

### 3.2 后端现状

当前 `backend/app/core/comic_chat_agent/agent_runner.py` 的本质，是一个标准 ReAct Agent：

- LLM 先输出 `thinking`
- 再生成工具调用计划
- 再发出 `tool_confirm / tool_start / tool_done`
- 再进入下一轮

这套机制本身没有问题，问题在于：

- **后端发的是执行事件**
- **前端却把事件原样当主内容展示**

也就是说，问题不主要在 Agent 能力，而在**前端没有做“任务层抽象”**。

### 3.3 根因判断

根因可以明确归纳为一句话：

> 当前系统把“Agent 内部执行事件”直接暴露成“用户主界面内容”，导致页面像调试台，而不像成熟 Agent 产品。

因此，本次改造的正确方向不是继续微调文案，而是：

- **在前端新增“任务视图层”**
- 把现有事件重新组织为：
  - 需求分析
  - 执行计划
  - 当前进度
  - 结果产物
  - 细节日志

## 四、改造后的信息架构

### 4.1 总体结构

新的 Agent 页面建议采用以下自上而下结构：

1. **任务头部区**
   - 用户原始需求
   - 需求分析
   - 任务类型
   - 风险级别
   - 当前阶段

2. **执行主线区**
   - 已规划步骤
   - 当前执行步骤
   - 已完成步骤
   - 待确认步骤

3. **结果产物区**
   - 图片
   - 视频
   - 音频
   - 下载入口
   - 结果版本对比

4. **助手结论区**
   - 当前阶段总结
   - 下一步建议
   - 是否需要继续优化

5. **执行细节区（折叠）**
   - thinking 明细
   - tool 参数
   - tool 原始结果
   - 调试日志

### 4.2 页面层级示意

```text
┌────────────────────────────────────┐
│ 任务头部区                         │
│ 用户需求                           │
│ 需求分析                           │
│ 当前阶段 / 总进度                  │
└────────────────────────────────────┘

┌────────────────────────────────────┐
│ 执行主线区                         │
│ Step 1 已完成                      │
│ Step 2 处理中                      │
│ Step 3 待确认                      │
└────────────────────────────────────┘

┌────────────────────────────────────┐
│ 结果产物区                         │
│ 图片 / 视频 / 音频 / 下载          │
└────────────────────────────────────┘

┌────────────────────────────────────┐
│ 助手结论区                         │
│ 当前总结 / 下一步建议              │
└────────────────────────────────────┘

┌────────────────────────────────────┐
│ 执行细节区（默认折叠）             │
│ thinking / tool 参数 / 原始日志    │
└────────────────────────────────────┘
```

这就是你要求的：

- **上面先是需求分析与任务主线**
- **下面才是思考细节和执行痕迹**

### 4.3 必须改变的默认规则

新的默认规则应当是：

- **默认展开：需求分析、执行计划、当前进度、结果产物**
- **默认折叠：thinking、工具参数、原始返回、冗长日志**

换句话说：

- 高层摘要默认可见
- 低层细节按需查看

而不是反过来。

## 五、核心交互改造方案

### 5.1 新增“任务头卡”

在每一轮用户发起任务后，前端不应只插入一条用户消息，而应生成一个 `taskCard`。

建议字段：

- `taskId`
- `userRequest`
- `intent`
- `analysis`
- `plan[]`
- `currentStep`
- `status`
- `artifacts[]`
- `logs[]`

展示内容：

- **原始需求**
- **需求分析**
- **任务拆解**
- **当前状态**

这样用户进入页面后，第一眼看到的永远是：

- 我要做什么
- Agent 怎么理解的
- Agent 准备怎么做

而不是零散消息。

### 5.2 新增“需求分析卡”

需求分析卡必须成为任务开始后的第一块内容。

固定模板建议：

- **任务类型**
  - 文生图 / 图像编辑 / 漫剧生成 / 图生视频 / 多步骤链式任务
- **目标理解**
  - 一句话概括用户真正想要的结果
- **约束识别**
  - 风格、人物一致性、是否需要审批、是否已有输入素材
- **执行策略**
  - 先做什么，再做什么，哪些步骤需要用户确认

例如：

- 任务类型：多步骤链式创作
- 目标理解：先生成仙侠退婚宴会图，再转视频，最后配旁白
- 约束识别：图生视频必须使用上一步图片绝对路径；TTS 文本固定
- 执行策略：生成图片 → 用户确认 → 图转视频 → 语音生成 → 汇总成片

这块内容应该固定在**任务头部最上方**。

### 5.3 新增“执行主线时间轴”

不要再把 `tool_start / tool_done / tool_confirm` 直接当普通聊天消息展示。

应该转成一个结构化时间轴：

- Step 1 需求分析 —— 已完成
- Step 2 生成首版图片 —— 已完成
- Step 3 图转视频 —— 处理中
- Step 4 语音合成 —— 待执行

状态值建议统一为：

- `pending`
- `running`
- `awaiting_approval`
- `completed`
- `failed`
- `canceled`

并配套颜色系统：

- 灰：待执行
- 蓝：处理中
- 黄：待确认
- 绿：已完成
- 红：失败

### 5.4 工具审批改造成“步骤确认”，不要像底层弹窗

当前的 `tool_confirm` 只是告诉用户“某工具待你确认”。

这不够产品化。

应该改为：

- 让用户确认的是“步骤动作”，而不是“底层工具名”
- 例如：
  - 不显示：`image_to_video`
  - 显示：`将已生成图片制作成动态视频`

确认卡应包含：

- 本步骤目标
- 输入来源
- 预期输出
- 大致耗时
- 风险说明
- 确认 / 取消

这样用户感知的是“创作流程推进”，而不是“系统在调 API”。

### 5.5 结果产物集中到独立区域

现在图片、视频、音频会分散挂在不同消息后面。

这很乱。

改造后应有统一的 `ArtifactsPanel`：

- 图片结果区
- 视频结果区
- 音频结果区
- 下载全部
- 复制路径
- 查看来源步骤

每个产物卡都应该标识：

- 来源步骤
- 生成时间
- 模型/工作流
- 是否为最终版

### 5.6 thinking 降级为“执行细节”

你提出“不要思考在下面，改动在上面，很奇怪”，本质上就是：

- 思考不应该成为主舞台
- 思考应该被降级为辅助信息

所以：

- `thinking` 不再作为主消息连续出现
- 它应该被收纳进：
  - “分析详情”
  - “执行细节”
  - “查看日志”

默认只显示摘要，例如：

- 已完成需求分析
- 已确定执行计划
- 已推进到第 2 步

点击后才展开原始 thinking 文本。

## 六、推荐的页面布局方案

### 6.1 最推荐：左右分栏布局

建议最终采用：

- **左侧：对话与任务主线**
- **右侧：任务详情与产物面板**

布局如下：

```text
┌──────────────────────┬──────────────────────┐
│ 左侧：主对话区       │ 右侧：任务侧栏       │
│                      │                      │
│ 用户消息             │ 需求分析             │
│ 助手结论             │ 执行计划             │
│ 当前步骤摘要         │ 当前进度             │
│ 审批卡               │ 结果产物             │
│                      │ 执行细节（折叠）     │
└──────────────────────┴──────────────────────┘
```

优点：

- 主线和细节分离
- 用户不用在聊天流里找进度
- 多媒体结果有稳定位置
- 更像成熟品牌 Agent

### 6.2 次优方案：单列分区布局

如果暂时不想做大布局改造，可以先做单列升级：

1. 任务头卡
2. 执行步骤区
3. 结果产物区
4. 对话区
5. 细节折叠区

即使不做左右分栏，也必须保证：

- **需求分析在最上面**
- **执行主线在思考前面**
- **结果区在细节前面**

## 七、结合当前代码的落地方案

### 7.1 前端改造重点

目标文件：

- `frontend/src/views/comic-agent/ComicAgentView.vue`
- `frontend/src/api/comic-agent.ts`

建议新增前端状态模型：

```ts
interface AgentTaskViewModel {
  id: string
  userRequest: string
  intent?: string
  analysis?: string
  plan: TaskStep[]
  currentStage?: string
  status: 'running' | 'completed' | 'failed' | 'awaiting_approval'
  artifacts: TaskArtifact[]
  logs: TaskLog[]
}
```

把现在的 `messages[]` 改成“双层结构”：

- `messages[]`：保留聊天内容
- `activeTask` / `taskList[]`：承载任务视图

不要再让所有事件直接渲染为消息。

### 7.2 事件重组规则

当前事件可以这样映射：

- `thinking`
  - 不直接生成主消息
  - 进入 `task.logs`
  - 同时抽取摘要更新 `analysis` 或 `currentStage`

- `tool_confirm`
  - 更新对应 `TaskStep.status = awaiting_approval`
  - 在主线区显示确认卡

- `tool_start`
  - 更新对应步骤 `running`
  - 把描述挂入主线步骤说明

- `tool_done`
  - 更新步骤为 `completed / failed`
  - 如果有媒体输出，写入 `artifacts`

- `text`
  - 作为“阶段结论”或“最终答复”进入聊天区

也就是说：

- 事件继续保留
- 但渲染时不再一股脑转成消息

### 7.3 后端配合建议

目标文件：

- `backend/app/core/comic_chat_agent/agent_runner.py`
- `backend/app/api/v1/comic_agent.py`

建议新增更高层的事件类型，而不是让前端自己猜：

- `task_analysis`
- `task_plan`
- `task_step_update`
- `task_artifact`
- `task_summary`

例如：

```json
{
  "type": "task_analysis",
  "intent": "multi_step_creation",
  "analysis": "用户希望先生成仙侠退婚宴会场景图，再转成动态视频，最后配旁白。"
}
```

```json
{
  "type": "task_step_update",
  "step_key": "generate_image",
  "title": "生成仙侠退婚宴会场景图",
  "status": "running"
}
```

这样前端就不必从 `thinking` 文本里猜“需求分析”是什么。

### 7.4 最关键的后端原则

后端不应该再把“展示语义”寄托在自然语言文案上。

应该把：

- 需求分析
- 计划
- 步骤状态
- 产物信息

都结构化输出。

这是从“提示词 UI”升级到“产品 UI”的关键一步。

## 八、推荐的视觉风格方向

### 8.1 整体气质

建议风格关键词：

- **专业**
- **克制**
- **任务导向**
- **高层摘要优先**
- **不过度拟人**

避免：

- 太多 emoji
- 太多彩色小标签同时出现
- 太像 IM 聊天软件

## 十、最终结论

### 10.1 设计结论

这次 Agent 改造的关键，不是继续润色几句提示文案，而是把当前系统从：

- **事件消息流 UI**

升级成：

- **任务主线 UI**

真正需要改变的是信息优先级：

- 以前：`thinking / tool / result` 混在一个流里
- 以后：`需求分析 → 执行计划 → 当前进度 → 结果产物 → 细节日志`

### 10.2 必须落实的硬规则

后续改造必须遵守下面 5 条硬规则：

1. **任务开始后的第一屏必须出现“需求分析”**
2. **执行计划必须在思考细节之前出现**
3. **结果产物必须集中展示，不能散落在消息流里**
4. **thinking 必须默认折叠，降级为细节层**
5. **工具名不应成为主展示对象，步骤目标才是主展示对象**

### 10.3 对你这句要求的直接响应

你说得非常对：

> “要就从上到下，不要思考在下面，改动在上面，很奇怪。第一件事应该是需求分析。”

这句话本质上已经给出了正确的产品原则：

- **先理解任务，再展示计划，再展示执行，再展示结果，最后才展示细节。**

这就是本次改造设计的总原则。
