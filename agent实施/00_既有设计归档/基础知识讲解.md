# 既有设计归档：基础知识讲解

## 1. 这个目录是什么

`00_既有设计归档` 是历史设计文档的副本仓库。

它保存项目演进过程中已经产生的重要分析和方案，作为后续实施的上下文来源。

```text
历史分析
  ↓
架构设计
  ↓
对比研究
  ↓
实施计划
  ↓
当前开发任务
```

## 2. 为什么要用归档目录

因为 Agent 项目上下文很长，涉及：

- 架构分析
- 竞品对比
- 设计方案
- TODO
- 已实现状态

如果没有归档，后续实施容易忘记之前为什么这样设计。

## 3. 好处

- **上下文集中**：不用在根目录到处找文档。
- **保留决策依据**：知道每个阶段为什么这样做。
- **避免重复讨论**：后续设计可以直接引用历史结论。
- **安全**：采用复制而非移动，不破坏原始文件引用。

## 4. 弊端

- **可能过时**：归档文档不一定反映最新代码。
- **可能重复**：同一结论可能出现在多个文档里。
- **需要维护**：重要变更后要补充新归档或更新索引。

## 5. 替代方案与对比

| 方案 | 优点 | 缺点 | 当前是否采用 |
|---|---|---|---|
| 根目录散放文档 | 简单 | 难维护、难检索 | 否 |
| 复制归档 | 安全、集中 | 有重复 | 是 |
| 移动原文档 | 最整洁 | 容易破坏引用 | 否 |
| 知识库/RAG | 可检索 | 实现成本高 | 后续可选 |

## 6. 图示

```text
根目录原文档         归档副本
──────────         ──────────
深度分析.md   ──→   00_既有设计归档/深度分析.md
优化设计.md   ──→   00_既有设计归档/优化设计.md
对比文档.md   ──→   00_既有设计归档/对比文档.md
```

## 7. 使用建议

- 新实施前先看归档里的相关设计。
- 不直接在归档文档上做实施记录。
- 新的实施记录写入 `99_实施记录`。
