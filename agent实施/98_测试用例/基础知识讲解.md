# 测试用例：基础知识讲解

## 1. 这个技术是什么

测试用例是对系统行为的可重复验证说明。

在 Agent 项目中，测试用例不仅测试代码是否运行，还要验证：

- 任务状态是否正确。
- 工具是否按顺序执行。
- 失败是否被正确识别。
- 前端是否展示正确状态。
- 最终报告是否可信。

## 2. 为什么要用测试用例

Agent 系统有不确定性：

```text
LLM 输出不稳定
工具可能失败
WebSocket 可能断开
前端可能误判状态
```

测试用例把期望行为写清楚，防止改一个模块破坏另一个模块。

## 3. 好处

- **防回归**：旧功能不被新改动破坏。
- **明确验收**：知道什么叫完成。
- **便于复盘**：失败时知道是哪类用例失败。
- **降低协作成本**：不同阶段有对应测试集。

## 4. 弊端

- **需要维护**：设计变了测试也要更新。
- **端到端测试成本高**：生成图片/视频耗时和花费资源。
- **LLM 随机性**：需要控制模型和输入。

## 5. 竞争技术 / 替代方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 手动点页面 | 直观 | 不可复现 |
| Markdown 测试用例 | 简单清晰 | 不能自动执行 |
| pytest 自动化 | 可重复 | 需要写代码 |
| Playwright E2E | 接近真实用户 | 实现成本高 |
| LangSmith Eval | Agent 评测强 | 外部依赖 |

## 6. 图示

```text
设计文档
  ↓
测试用例
  ↓
实施代码
  ↓
运行验证
  ↓
实施日志记录结果
```

## 7. 关键点

每个阶段实施前先更新测试用例。

```text
没有测试用例，不开始重要改造
```
