# Soul AI Lab 在线 Demo → ttsapp 迁移可行性分析

> 目标：将 HuggingFace Spaces 上的 Soul AI Lab 在线 Demo 以 API 方式集成到 ttsapp 后端，  
> 使前端可通过 ttsapp 统一接口调用数字人生成、歌声合成、播客语音、全双工对话等能力。

---

## 一、ttsapp 现有架构概览

```
前端 (Vue 3 + Element Plus)
    ↓ axios
后端 (FastAPI :8000)
    ├─ /api/v1/tts      → FishSpeechClient (httpx) → Fish Audio API
    ├─ /api/v1/asr       → FishSpeechClient (httpx) → Fish Audio API
    ├─ /api/v1/chat      → SouthgridLLMClient (httpx) → 南方电网 AI Gateway
    └─ /api/v1/voice-models → FishSpeechClient → Fish Audio API
```

**核心模式**: 后端作为**统一代理层**，用 `httpx` 转发请求到外部 API。  
新增 Soul 模型集成只需按此模式添加新的 Client + Router。

---

## 二、HuggingFace Spaces 的 API 调用机制

### 关键发现：Gradio 应用天然暴露 REST API

所有基于 Gradio 构建的 HuggingFace Spaces **自动提供两种 API 调用方式**：

### 方式 1：`gradio_client` Python SDK（推荐）

```python
from gradio_client import Client, handle_file

# 连接到 Space
client = Client("Soul-AILab/SoulX-FlashHead")

# 调用 API（自动排队、自动重试）
result = client.predict(
    image=handle_file("reference.jpg"),
    audio=handle_file("speech.wav"),
    api_name="/generate"
)
# result → 生成的视频文件路径
```

**优点**:
- 官方维护，稳定可靠
- 自动处理排队、文件上传/下载、WebSocket 通信
- 支持异步：`async with AsyncClient(...)`
- 自动发现所有可用 API 端点

### 方式 2：REST API 直接调用（httpx 兼容）

每个 Gradio Space 暴露 REST 端点：
```
POST https://{space-id}.hf.space/api/predict
POST https://{space-id}.hf.space/call/{api_name}
GET  https://{space-id}.hf.space/call/{api_name}/{event_id}
```

```python
import httpx

# 第一步：提交任务
resp = await client.post(
    "https://soul-ailab-soulx-flashhead.hf.space/call/generate",
    json={"data": [image_base64, audio_base64]},
)
event_id = resp.json()["event_id"]

# 第二步：轮询/SSE 获取结果
async with client.stream("GET",
    f"https://soul-ailab-soulx-flashhead.hf.space/call/generate/{event_id}"
) as stream:
    async for line in stream.aiter_lines():
        # 解析 SSE 格式结果
        ...
```

## 三、各模型迁移可行性评估

### 3.1 SoulX-FlashHead（数字人视频生成）

| 维度 | 评估 |
|------|------|
| **可行性** | ✅ **高** |
| **API 端点** | `https://soul-ailab-soulx-flashhead.hf.space/api/predict` |
| **输入** | 参考图片 (image) + 音频 (audio) |
| **输出** | 视频文件 (mp4) |
| **延迟** | 排队 + 生成，预计 30s~5min（取决于队列长度） |
| **与 ttsapp 集成** | 新建 `app/core/flashhead_client.py`，模式与 `fish_speech.py` 一致 |

**集成示意**:
```python
# app/core/flashhead_client.py
from gradio_client import Client, handle_file
import tempfile, os

class FlashHeadClient:
    def __init__(self):
        self.space_id = "Soul-AILab/SoulX-FlashHead"

    async def generate_video(self, image_bytes: bytes, audio_bytes: bytes) -> bytes:
        client = Client(self.space_id)
        # 保存临时文件
        with tempfile.NamedTemporaryFile(suffix=".jpg", delete=False) as img_f:
            img_f.write(image_bytes)
            img_path = img_f.name
        with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as aud_f:
            aud_f.write(audio_bytes)
            aud_path = aud_f.name
        try:
            result = client.predict(
                image=handle_file(img_path),
                audio=handle_file(aud_path),
                api_name="/generate"
            )
            with open(result, "rb") as f:
                return f.read()
        finally:
            os.unlink(img_path)
            os.unlink(aud_path)
```

**风险**:
- ZeroGPU 排队制，高峰期延迟不可控
- 单次生成视频长度有限制（Demo 通常限制 10-30s）
- 无 SLA 保障，Space 可能随时 sleep

---

### 3.2 SoulX-Singer（歌声合成 + SVC）

| 维度 | 评估 |
|------|------|
| **可行性** | ✅ **高** |
| **API 端点** | `https://soul-ailab-soulx-singer.hf.space/api/predict` |
| **输入** | 参考音频 + 歌词 + MIDI/F0 + 控制模式 |
| **输出** | 音频文件 (wav) |
| **延迟** | 30s~3min |
| **与 ttsapp 集成** | 新建 `app/core/singer_client.py` |

**两个子功能**:
1. **歌声合成 (SVS)**: 参考音色 + 歌词 + 旋律 → 生成歌声
2. **歌声转换 (SVC)**: 参考音色 + 源歌曲 → 转换音色

**集成与 ttsapp TTS 模块天然互补**：ttsapp 现有 TTS 是"说话"，Singer 补充"唱歌"能力。

---

### 3.3 SoulX-Podcast（播客语音合成）

| 维度 | 评估 |
|------|------|
| **可行性** | ✅ **最高（最适合集成）** |
| **API 端点** | `https://soul-ailab-soulx-podcast-1-7b.hf.space/api/predict` |
| **输入** | 参考音频 + 文本 + 方言选项 |
| **输出** | 音频文件 (wav) |
| **延迟** | 10s~2min |
| **与 ttsapp 集成** | 新建 `app/core/podcast_client.py` |

**为什么最适合**:
- **功能定位与 ttsapp 高度重叠**: 都是文字转语音
- **差异化价值**: Podcast 支持多人对话、方言、副语言（笑声/叹气），Fish Audio 不具备这些能力
- **可作为 TTS 引擎的补充选项**: 前端 TTS 页面新增"引擎选择"下拉框 → Fish Audio / SoulX-Podcast

---

### 3.4 SoulX-Duplug（全双工语音对话）

| 维度 | 评估 |
|------|------|
| **可行性** | ⚠️ **中（需额外工作）** |
| **入口** | `https://soulx-duplug.sjtuxlance.com/`（非 Gradio，自定义 WebSocket） |
| **协议** | WebSocket 流式音频双向传输 |
| **延迟** | 实时 |
| **与 ttsapp 集成** | 需 WebSocket 代理，复杂度较高 |

**难点**:
- 不是 Gradio Space，无法使用 `gradio_client`
- 需要逆向分析其 WebSocket 协议
- 实时音频流转发对服务器带宽有要求
- 第三方服务器可用性不可控

**建议**: 暂缓集成，如需全双工能力可本地部署 Duplug 模型。

### 4.3 通用基类设计

```python
# app/core/gradio_base.py
from gradio_client import Client, handle_file
import tempfile, os, asyncio
from loguru import logger

class GradioSpaceClient:
    """HuggingFace Gradio Space 通用调用基类"""

    def __init__(self, space_id: str, timeout: int = 300):
        self.space_id = space_id
        self.timeout = timeout

    def _get_client(self) -> Client:
        return Client(self.space_id)

    async def _save_temp(self, data: bytes, suffix: str) -> str:
        with tempfile.NamedTemporaryFile(suffix=suffix, delete=False) as f:
            f.write(data)
            return f.name

    async def _call(self, api_name: str, **kwargs):
        """在线程池中调用 Gradio（同步SDK）以避免阻塞事件循环"""
        loop = asyncio.get_event_loop()
        def _sync_call():
            client = self._get_client()
            return client.predict(api_name=api_name, **kwargs)
        return await loop.run_in_executor(None, _sync_call)
```

## 五、风险评估与应对

### 5.1 核心风险

| 风险 | 严重度 | 概率 | 应对 |
|------|--------|------|------|
| **HF Space 排队延迟不可控** | 🔴 高 | 高 | 异步任务 + 前端轮询（复用现有 TTSTask 模式） |
| **HF Space 被 sleep/下线** | 🔴 高 | 中 | 健康检查 + 自动唤醒 + 前端优雅降级提示 |
| **ZeroGPU 配额限制** | 🟡 中 | 中 | 实现请求排队 + 速率限制（Redis 令牌桶） |
| **API 接口变更** | 🟡 中 | 低 | `gradio_client` 自动发现端点，适应性强 |
| **文件上传/下载带宽** | 🟡 中 | 中 | 视频/音频压缩 + CDN 缓存 |
| **无 SLA 保障** | 🔴 高 | 确定 | 仅用于体验/测试，生产环境需本地部署 |

### 5.2 免费 vs 本地部署对比

| 方案 | 成本 | 延迟 | 可靠性 | 并发 | 适用场景 |
|------|------|------|--------|------|---------|
| **HF Space 转发** | ✅ 免费 | ❌ 30s~5min | ❌ 无保障 | ❌ 1-2 并发 | 体验/演示/个人使用 |
| **本地部署 FlashHead Lite** | 🟡 需 RTX 4090 | ✅ <1s | ✅ 可控 | ✅ 3路并发/卡 | 生产环境 |
| **HF Inference Endpoints** | 🔴 付费 ($1-5/h) | ✅ 稳定 | ✅ SLA | ✅ 可扩展 | 商用生产 |

## 六、实施计划

### Phase 1：Podcast 集成（最快见效，1-2 天）

> 与 ttsapp 现有 TTS 功能最互补，改动最小

1. 安装 `gradio_client`
2. 新建 `podcast_client.py` + `podcast.py` 路由
3. 前端 TTS 页面新增"引擎选择": Fish Audio / SoulX-Podcast
4. 复用现有异步任务流程（TTSTask 表）

### Phase 2：Singer 集成（2-3 天）

1. 新建 `singer_client.py` + `singing.py` 路由
2. 前端新增"AI 歌声"页面
3. 支持 SVS + SVC 两种模式

### Phase 3：FlashHead 集成（3-5 天）

1. 新建 `flashhead_client.py` + `digital_human.py` 路由
2. 前端新增"数字人"页面：上传照片 + 音频 → 生成视频
3. 异步任务 + 进度轮询
4. 视频存储与播放

### Phase 4（可选）：本地部署替代

> 当在线方案无法满足延迟/并发需求时

1. 本地部署 FlashHead Lite（单卡 4090）
2. 本地部署 Podcast（vLLM + Docker）
3. 将 `gradio_client` 调用替换为本地 HTTP 接口，**路由层无需改动**

## 七、结论

### ✅ 可行，且实施成本低

| 结论 | 理由 |
|------|------|
| **技术上完全可行** | Gradio Space 天然暴露 API，`gradio_client` SDK 官方支持 |
| **与现有架构兼容** | ttsapp 已有"后端代理外部 API"的成熟模式（FishSpeechClient），新增模型只需复制此模式 |
| **改动量小** | 每个模型仅需新增 1 个 Client + 1 个 Router + 1 个前端页面 |
| **渐进式集成** | 可按优先级分阶段集成，不影响现有功能 |
| **零成本启动** | 利用 HF 免费 ZeroGPU，无需额外硬件 |

### ⚠️ 限制

| 限制 | 说明 |
|------|------|
| **不适合生产环境** | HF Space 无 SLA，排队延迟不可控 |
| **仅适合体验/演示** | 并发有限，高峰期可能等待数分钟 |
| **生产方案** | 需本地部署模型或使用 HF Inference Endpoints（付费） |

### 📋 推荐行动

1. **立即可做**: 集成 SoulX-Podcast 作为 TTS 补充引擎（1-2 天）
2. **短期**: 集成 Singer + FlashHead，丰富平台能力（1 周）
3. **中期**: 有 GPU 后本地部署 FlashHead Lite，替换 HF Space 调用（无需改路由层）
