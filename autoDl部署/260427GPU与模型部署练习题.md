### 【GPU 架构与计算能力】

**1.** NVIDIA GPU 的"计算能力"（Compute Capability）是用来描述什么的？

- A. GPU 的显存带宽大小
- B. GPU 支持的 CUDA 指令集版本
- C. GPU 的 FP32 运算速度
- D. GPU 的核心数量

> **答案：B** — SM（Streaming Multiprocessor 版本号）决定支持哪些 CUDA 指令集和特性，不直接代表速度或显存。

---

**2.** 以下 GPU 按"SM 版本从低到高"排序，正确的是？

- A. V100 → A100 → H100 → RTX 5090
- B. RTX 5090 → H100 → A100 → V100
- C. A100 → V100 → H100 → RTX 5090
- D. H100 → RTX 5090 → A100 → V100

> **答案：A** — V100(sm70) < A100(sm80) < H100(sm90) < RTX 5090(sm120)，按发布时间和架构代际递增。

---

**3.** RTX 5090 的计算能力版本是 SM 12.0，A100 是 SM 8.0。如果某个 AI 模型要求 `torch==2.2.0`，该版本 torch 最高支持到 SM 8.9。请问这个模型能在 RTX 5090 上运行吗？

- A. 能，SM 12.0 > 8.9，向上兼容
- B. 不能，torch 2.2.0 不支持 SM 12.0
- C. 能，只要 CUDA 版本匹配就行
- D. 能，但速度会比 A100 慢

> **答案：B** — GPU 兼容性是「向下」而非「向上」：新 GPU 需要新版 torch 才能支持。torch 2.2.0 没有编译 sm120 的内核，运行时会报 `no kernel image is available`。

---

**4.** flash_attn 加速库在 RTX 5090（SM 12.0）上安装失败的根本原因是？

- A. 显存不够大
- B. CUDA 版本太新
- C. 没有针对 SM 12.0 的预编译安装包，源码编译也失败
- D. flash_attn 只支持 A100

> **答案：C** — flash_attn 预编译 wheel 最高到 sm90（H100）。在 RTX 5090 上源码编译同样因架构不在支持列表内而失败。

---

**5.** A100 SXM4 相比 RTX 5090 的优势是什么？（多选）

- A. FP16 算力更高
- B. 兼容 torch 2.2.0（Singer 所需）
- C. 支持 flash_attn 预编译安装
- D. 显存更大（40GB vs 32GB）

> **答案：B C D** — A100 FP16 算力约 312 TFLOPS，RTX 5090 约 837 TFLOPS，所以 A 错；但 A100(sm80) 兼容 torch 2.2.0、有 flash_attn 预编译包、40GB > 32GB，B/C/D 均正确。

**6.** CUDA JIT（Just-In-Time）编译是指什么？

- A. 在 GPU 上实时训练模型
- B. PyTorch 第一次运行某操作时，现场为当前 GPU 编译 CUDA 内核代码
- C. 把 Python 代码编译成 C++ 以提速
- D. 自动选择最优的 GPU 核心数量

> **答案：B** — JIT（Just-In-Time）编译是「运行时编译」。PyTorch 发现当前 GPU 没有缓存的内核时，现场编译并缓存，所以第一次慢、后续快。

**7.** 本次部署中，FlashHead 第一次推理 chunk-0 耗时 **109 秒**，后续每个 chunk 只需 **0.18 秒**，这个现象最可能的原因是？

- A. 第一次需要从磁盘加载模型权重
- B. 第一次触发 CUDA JIT 编译，编译完成后内核被缓存
- C. 网络延迟导致第一次请求慢
- D. Python 垃圾回收导致暂停

> **答案：B** — 实测 chunk-0 耗时 109.73s，chunk-1~9 各约 0.18s，600:1 的比例正是 JIT 编译的典型特征，与模型加载无关（模型已在此之前加载完毕）。

### 【存储体系】

**8.** AI 推理时，模型权重需要加载到哪里才能被 GPU 计算？

- A. 数据盘（SSD）
- B. 系统内存（RAM）
- C. 显存（VRAM）
- D. CPU 缓存

> **答案：C** — GPU 的计算单元只能直接访问显存（VRAM）。模型权重需要先从磁盘 → 内存 → 显存，才能被 GPU 计算。

---

**9.** 以下四种存储，访问速度从快到慢排列正确的是？

- A. 显存 > 内存 > 数据盘 > 系统盘
- B. 内存 > 显存 > 系统盘 > 数据盘
- C. 显存 > 内存 > 系统盘 ≈ 数据盘
- D. 显存 > 数据盘 > 内存 > 系统盘

> **答案：C** — 显存带宽 ~3 TB/s，内存 ~100 GB/s，NVMe SSD（系统盘/数据盘）~7 GB/s，两者速度相近。

**10.** FlashHead 模型文件在磁盘上占 14 GB，但实际推理时显存只用了约 11 GB，为什么？

> **答案：B** — FlashHead 仓库含大量演示 MP4/PNG 文件；加载时框架自动用 FP16，比磁盘 FP32 节省约一半显存。

**11.** AutoDL 实例的数据盘（`/root/autodl-tmp`）和系统盘的最大区别是？

**25.** 通过 SSH 执行一个需要运行 1 小时的下载任务，中途 SSH 连接断开，以下说法正确的是？

- A. 下载任务会继续运行到完成

**思考题 1：依赖冲突的架构决策**

你接手一个项目，包含两个服务：
- 服务 A 要求 `torch==2.7.1`，`transformers==4.57.3`
- 服务 B 要求 `torch==2.2.0`，`numpy<2.0`

有人提议用一个 conda 环境运行两个服务以节省磁盘。

**请回答**：这个方案可行吗？为什么？应该如何正确设计？如果两个服务的 torch 版本不同，会带来哪些具体的运行时问题？

> **▶ 参考答案**
>
> **不可行。** pip 后安装的 torch 版本会直接覆盖前者，导致其中一个服务必然崩溃。即使勉强共存，torch 版本锁定后，依赖它的 transformers/numpy 版本约束也会相互破坏（如 `numpy<2.0` 与新版 transformers 要求的 `numpy>=2.0` 直接冲突）。
>
> **正确设计**：3 个独立 conda 环境（`--prefix` 指向数据盘），各自独立安装、各自激活后启动服务：
> ```bash
> conda create --prefix /root/autodl-tmp/envs/env-a python=3.10
> conda create --prefix /root/autodl-tmp/envs/env-b python=3.10
> ```
> 运行时问题：`ImportError` 因 API 变更；算子 shape 不匹配静默计算错误；CUDA 内核版本不一致导致段错误。

**思考题 2：GPU 选型决策**

你要部署一个 AI 服务，需要同时运行三个模型：
- 模型 A：要求 `torch==2.2.0`，推理显存 ~6 GB
- 模型 B：要求 `torch==2.7.1`，推理显存 ~11 GB
- 模型 C：要求 `torch==2.7.1`，常驻显存 ~5 GB

候选 GPU：
| GPU | 显存 | SM | 价格 |
|---|---|---|---|
| RTX 5090 | 32 GB | SM 12.0 | ¥5/h |
| A100 40GB | 40 GB | SM 8.0 | ¥5/h |
| RTX 4090 | 24 GB | SM 8.9 | ¥3/h |

**请回答**：应该选哪块 GPU？给出分析过程，包括显存计算、SM 兼容性验证。

> **▶ 参考答案**
>
> **应选 A100 40GB（SM 8.0）。**
>
> | GPU | SM 兼容性 | 显存是否够（峰值 6+11+5=22 GB） | 结论 |
> |---|---|---|---|
> | RTX 5090 | SM 12.0 **不支持** torch 2.2.0 → 模型 A 无法运行 | — | ❌ |
> | RTX 4090 | SM 8.9 ✅ | 24 GB，峰值 22 GB 风险极高 | ❌ 不推荐 |
> | A100 40GB | SM 8.0 ✅ | 40 GB，峰值 22 GB / 40 GB = 55% | ✅ |
>
> A100 同时满足 torch 2.2.0（sm80 ≥ sm80）、torch 2.7.1、flash_attn 预编译、显存充足四项要求。

**思考题 3：磁盘规划**

一个同事计划在 50 GB 数据盘上部署项目，他的预估：
- 模型 A：参数量 1.3B，估算 FP16 大小 = 1.3 × 10⁹ × 2 = 2.6 GB
- 模型 B：参数量 1.7B，估算 = 3.4 GB
- 合计：约 6 GB，50 GB 绰绰有余

结果实际下载后，模型 A 占了 14 GB，模型 B 占了 5.4 GB。

**请回答**：
1. 为什么估算与实际相差这么大？列出至少 3 个原因

**思考题 11：CUDA 版本 vs SM 版本的区别**

同事说："我装的是 CUDA 12.8，GPU 是 RTX 5090，CUDA 够新所以 torch 2.2.0 应该能跑。" 但实际上跑不了。

>
> 3. 查看 torch 编译了哪些架构：
>    ```python
>    import torch
>    print(torch.cuda.get_arch_list())
>    # 示例输出: ['sm_70', 'sm_75', 'sm_80', 'sm_86', 'sm_89']
>    ```

**思考题 12：模型量化的原理与权衡**

>    ```python
>    model = AutoModel.from_pretrained(

>    )
>    ```

**思考题 18：服务日志与故障排查**

FlashHead 服务在某次请求后无响应，`tmux` 窗口显示没有新的输出，但进程还活着（`ps aux` 能看到）。

**请回答**：
1. 进程存活但无响应，可能的原因有哪些？
2. 如何判断是 GPU 死锁、CPU 阻塞还是 Python 死锁？写出具体诊断命令
3. 如何设计一个"服务无响应自动重启"的守护脚本？给出核心逻辑

> **▶ 参考答案**
>
> 1. **可能原因**：① Python GIL 死锁（线程互相等待）；② CUDA 算子挂起（GPU kernel 无限等待）；③ 等待 I/O（磁盘/网络读取卡住）；④ Gradio 请求队列满（并发请求超出处理能力）。
>
> 2. **诊断命令**：
>    ```bash
>    # GPU 状态（看是否有 Volatile GPU-Util 持续 100% 或 0%）
>    nvidia-smi -l 1
>    # Python 线程栈（找死锁点）
>    kill -SIGUSR1 <PID>    # 触发 Python 打印所有线程栈
>    # 系统调用追踪
>    strace -p <PID> -e trace=futex   # 看是否卡在锁等待
>    ```
>
> 3. **守护脚本核心逻辑**：
>    ```bash
>    while true; do
>      # 用 curl 发一个健康检查请求
>      if ! curl -sf http://localhost:6006/ > /dev/null 2>&1; then
>        echo "$(date): 服务无响应，重启中..." >> ~/logs/watchdog.log
>        kill -9 $(lsof -ti:6006) 2>/dev/null
>        bash start_flashhead.sh &
>      fi
>      sleep 30
>    done
>    ```

**思考题 19：异步推理 vs 同步推理的架构选择**

你的 FlashHead 服务每次生成视频需要 30 秒，Podcast 服务每次合成音频需要 10 秒。前端用户直接调用 HTTP API 等待结果。

**请回答**：
1. 同步接口（等待 30 秒返回）有什么问题？
2. 如何设计异步任务队列架构？描述核心流程
3. 在不引入额外中间件的情况下，用 Python 标准库如何实现简单的任务队列？

> **▶ 参考答案**
>
> 1. **同步接口的问题**：① HTTP 超时（Nginx/浏览器默认 30~60 秒超时，接近临界）；② 阻塞服务器线程，并发请求时后续请求全部排队；③ 用户体验差，无法看到进度。
>
> 2. **异步任务队列架构**：
>    ```
>    前端 POST /generate → 后端立即返回 task_id
>    ↓
>    后端将任务放入队列（Redis/内存队列）
>    ↓
>    Worker 进程消费队列，执行推理（30s）
>    ↓
>    前端轮询 GET /task/{task_id}/status → pending/running/done
>    ↓
>    完成后返回结果 URL
>    ```
>
> 3. **Python 简单实现（`queue.Queue` + `threading`）**：
>    ```python
>    import queue, threading, uuid
>    task_queue = queue.Queue()
>    task_results = {}
>
>    def worker():
>        while True:
>            task_id, params = task_queue.get()
>            task_results[task_id] = run_inference(params)
>
>    threading.Thread(target=worker, daemon=True).start()
>    ```

**思考题 20：从"部署成功"到"生产可靠"的差距**

你完成了首次部署，服务在 AutoDL 上运行正常。但真正"生产可靠"还差很远。

**请回答**：
1. 列举至少 5 个"首次跑通"但"生产不够用"的维度
2. 在云 GPU 平台上，哪些故障是平台侧的（你无法控制），哪些是代码侧的（你可以优化）？
3. 如果你只有 1 天时间提升稳定性，优先级最高的 3 件事是什么？说明理由

> **▶ 参考答案**
>
> 1. **5 个差距维度**：
>    - **自动重启**：服务崩溃后没有守护进程自动拉起
>    - **日志缺失**：服务输出只在 tmux 里，没有持久化日志文件，故障后无法追溯
>    - **健康检查**：没有监控端点，外部无法感知服务是否存活
>    - **资源隔离**：GPU OOM 会杀掉整个服务，无内存限制保护
>    - **开机自启**：每次服务器重启都需要手动执行启动脚本
>
> 2. **平台侧故障**（不可控）：服务器宕机、网络波动、GPU 硬件故障。**代码侧**（可优化）：内存泄漏、未处理异常导致崩溃、端口未清理、日志缺失。
>
> 3. **优先级最高的 3 件事**：
>    - ① **持久化日志**（`2>&1 | tee ~/logs/service.log`）：故障后能看到错误信息，成本极低、价值极高
>    - ② **守护重启脚本**（`watchdog.sh` 每 30 秒 curl 健康检查）：防止服务崩溃后无人知晓
>    - ③ **幂等启动脚本**（`start_all.sh` 先 kill 旧进程再启动）：避免"端口占用"导致的重启失败
