## 一、GPU vs CPU：为什么 GPU 适合 AI？

| 对比维度 | CPU | GPU |
|---|---|---|
| 核心数量 | 8~64 个大核心 | 数千~万个小核心 |
| 设计目标 | 低延迟、复杂逻辑 | 高吞吐、并行计算 |
| 擅长任务 | 串行、分支密集 | 矩阵乘法、向量运算 |
| 典型代表 | Intel i9, AMD 7950X | NVIDIA A100, RTX 4090 |

AI 模型推理的核心操作是**大规模矩阵乘法**（每个 Transformer 层都是矩阵乘），这正是 GPU 的强项。

## 二、GPU 内部架构

### SM（Streaming Multiprocessor，流式多处理器）

SM 是 GPU 的基本计算单元，类似 CPU 的"核心"。每个 SM 包含：

```
┌─────────────────────────────────┐
│       Streaming Multiprocessor  │
│  ┌──────────┐  ┌─────────────┐  │
│  │ CUDA Core│  │ Tensor Core │  │
│  │（FP32运算）│  │（矩阵加速）  │  │
│  └──────────┘  └─────────────┘  │
│  ┌──────────────────────────┐   │
│  │  L1 Cache / Shared Memory│   │
│  └──────────────────────────┘   │
│  ┌──────────────────────────┐   │
│  │  Registers（寄存器文件）  │   │
│  └──────────────────────────┘   │
└─────────────────────────────────┘
```

- **CUDA Core**：基础浮点运算单元（FP32/FP64）
- **Tensor Core**：专为矩阵乘法设计，FP16/BF16/INT8 速度远超 CUDA Core，AI 推理主要用这个
- **SM 数量越多，并行能力越强**

### 主要 GPU 的 SM 配置

| GPU | SM 数量 | Tensor Core | 显存 | SM 版本 |
|---|---|---|---|---|
| RTX 3090 | 82 | 328 | 24 GB | sm86 |
| RTX 4090 | 128 | 512 | 24 GB | sm89 |
| A100 40GB | 108 | 432 | 40 GB HBM2e | sm80 |
| A100 80GB | 108 | 432 | 80 GB HBM2e | sm80 |
| H100 | 132 | 528 | 80 GB HBM3 | sm90 |
| RTX 5090 | 170 | 680 | 32 GB | sm120 |

## 三、SM 版本（Compute Capability）与 CUDA 兼容性

### SM 版本是什么？

SM 版本（也叫 Compute Capability，计算能力）是 NVIDIA 对 GPU **硬件架构**的编号：

```
SM 版本  GPU 架构名   代表 GPU
sm70    Volta       V100
sm80    Ampere      A100
sm86    Ampere      RTX 3090
sm89    Ada         RTX 4090
sm90    Hopper      H100
sm120   Blackwell   RTX 5090
```

**关键规则：SM 版本决定了 GPU 能运行哪些预编译 CUDA 内核。**

### PyTorch 与 SM 版本的关系

PyTorch 在发布时，会**提前编译**好支持到某个 SM 版本的 CUDA 内核（kernel）：

```
torch 2.2.0  →  编译支持最高到 sm89（RTX 4090）
torch 2.7.0  →  编译支持最高到 sm120（RTX 5090）
```

如果你的 GPU 的 SM 版本比 torch 支持的最高版本还新，就会报错：

```
RuntimeError: CUDA error: no kernel image is available for execution on the device
```

**验证命令**：

```python
import torch
print(torch.cuda.get_arch_list())   # 查看 torch 支持的 SM 列表
print(torch.version.cuda)           # torch 编译时用的 CUDA 版本
```

### CUDA 版本 vs SM 版本

这是两个独立概念，**不要混淆**：

| 概念 | 含义 | 示例 |
|---|---|---|
| **CUDA 版本** | 驱动/工具链的 API 版本 | CUDA 12.8（驱动版本） |
| **SM 版本** | GPU 硬件架构代号 | sm80（A100 的架构） |

CUDA 12.8 驱动可以管理 sm70~sm120 的各种 GPU，但 torch 2.2.0 内核只编译到 sm89，因此 CUDA 12.8 + torch 2.2.0 + RTX 5090（sm120）= **无法运行**。

## 四、内存体系：从磁盘到 GPU 运算

GPU 只能直接访问**自己的显存（VRAM）**，模型数据必须经历完整的搬运链路：

```
磁盘（SSD）
    │  读取速度：~7 GB/s（NVMe）
    ▼
系统内存（RAM）
    │  传输速度：~16 GB/s（PCIe 4.0 x16）
    ▼
GPU 显存（VRAM）
    │  带宽：~2 TB/s（A100 HBM2e）
    ▼
SM 寄存器 / L1 Cache（最快，KB 级）
    ▼
       GPU 计算
```

### 各层的作用

| 存储层 | 容量（典型） | 速度 | 持久性 | 作用 |
|---|---|---|---|---|
| 磁盘（数据盘）| 50~500 GB | 7 GB/s | 永久 | 存放模型文件、代码、数据集 |
| 系统内存（RAM）| 32~256 GB | 16 GB/s | 重启后清空 | 加载模型时的中转、CPU 运算 |
| GPU 显存（VRAM）| 16~80 GB | ~2 TB/s | 关机后清空 | 存放权重、激活值、KV Cache |
| GPU L2 Cache | ~40~50 MB | ~10 TB/s | 极短暂 | SM 共享缓存，减少显存访问 |
| SM 寄存器 | ~256 KB/SM | ~20 TB/s | 极短暂 | 当前计算的中间值 |

### 带宽差距的实际影响

显存带宽（~2 TB/s）远超 PCIe 传输（~16 GB/s），因此：
- 一旦模型加载到显存，推理速度极快
- **瓶颈在于首次从磁盘加载**，14 GB 模型需要约 2 秒（NVMe）才能装入显存

## 五、如果模型需要 50 GB 显存，磁盘和内存需要多少？

### 磁盘需求

模型文件在磁盘上通常以 **FP32**（4字节/参数）存储，推理时以 **FP16**（2字节/参数）加载：

```
VRAM 需求 50 GB（FP16）
↓
磁盘文件大小 ≈ 50 × 2 = 100 GB（FP32）
```

但如果发布者已经以 FP16 格式发布（越来越常见），磁盘也是 50 GB。  
**实际操作**：看 HuggingFace 仓库的 Files 页面确认实际文件大小。

### 系统内存（RAM）需求

模型从磁盘 → RAM → VRAM 的过程：

```
情况 A（PyTorch 默认行为）：
  磁盘文件 100 GB → 先完整读入 RAM → 再转移/转换到 VRAM
  RAM 需求 ≥ 100 GB + 系统开销

情况 B（内存映射加载，mmap）：
  不把整个文件读入 RAM，按需映射
  RAM 需求 ≈ 50 GB（VRAM 大小）+ OS 开销

情况 C（使用 accelerate 的 low_cpu_mem_usage）：
  model = AutoModel.from_pretrained("...", low_cpu_mem_usage=True)
  RAM 峰值可降低到 VRAM 大小的一半左右
```

**实际建议**：RAM 至少等于 VRAM 需求（50 GB），**推荐 2 倍**（100 GB）以避免 OOM 崩溃。AutoDL A100 实例通常配备 80~256 GB RAM，足够应对。

## 六、数据类型：FP32 / FP16 / BF16 / INT8 / INT4

### 各精度对比

| 类型 | 位宽 | 字节 | 数值范围 | 精度 | 适用场景 |
|---|---|---|---|---|---|
| FP32 | 32 bit | 4 B | ±3.4×10³⁸ | 高 | 训练、磁盘存储 |
| FP16 | 16 bit | 2 B | ±65504 | 中 | 推理首选 |
| BF16 | 16 bit | 2 B | ±3.4×10³⁸ | 中 | A100/H100 推理 |
| INT8 | 8 bit | 1 B | -128~127 | 低 | 量化推理 |
| INT4 | 4 bit | 0.5 B | -8~7 | 极低 | 极限压缩推理 |

### FP16 vs BF16

- **FP16**：精度更高，但数值范围小，容易溢出（推理过程中梯度爆炸问题）
- **BF16**：精度略低，但数值范围与 FP32 相同，不易溢出，A100/H100 原生支持，**LLM 推理推荐 BF16**

```python
model = AutoModel.from_pretrained("...", torch_dtype=torch.bfloat16)  # 推荐
model = AutoModel.from_pretrained("...", torch_dtype=torch.float16)   # 备选
```

### 磁盘 FP32 → 显存 FP16 的原理

```
磁盘上存储：                  显存中加载：
┌──────────┐                ┌──────────┐
│权重: FP32│  框架自动转换   │权重: FP16│
│4字节/参数 │ ─────────────▶ │2字节/参数 │
│文件: 28GB│                │显存: 14GB│
└──────────┘                └──────────┘
```

这就是为什么磁盘文件比实际推理显存大约一倍。

## 七、显存估算公式

### 基础估算（推理）

```
显存(GB) = 参数量(B) × 精度字节数 × 1.2（框架开销系数）

FP16：参数量(B) × 2 × 1.2
BF16：参数量(B) × 2 × 1.2
INT8：参数量(B) × 1 × 1.2

示例：
  7B 模型 FP16  → 7 × 2 × 1.2 = 16.8 GB
  7B 模型 INT8  → 7 × 1 × 1.2 = 8.4 GB
  7B 模型 INT4  → 7 × 0.5 × 1.2 = 4.2 GB
```

### 额外显存开销

| 来源 | 大小估算 |
|---|---|
| KV Cache（长文本推理）| batch_size × seq_len × layers × heads × 2字节，可达数 GB |
| 激活值（前向传播）| 约为模型权重的 10~30% |
| CUDA 框架开销 | 约 500 MB~2 GB |
| flash_attn 优化后 | KV Cache 开销大幅减少（核心优势） |

### 实际验证

```python
import torch
# 查看当前显存占用
print(f"已分配: {torch.cuda.memory_allocated()/1e9:.2f} GB")
print(f"已预留: {torch.cuda.memory_reserved()/1e9:.2f} GB")
```

```bash
# 命令行实时监控
nvidia-smi
watch -n1 nvidia-smi   # 每秒刷新
```

## 八、FlashHead 仓库文件 vs 显存的关系

以 FlashHead（1.3B 参数）为例：

```
HuggingFace 仓库总大小：14 GB
├── 模型权重文件（FP32）：约 5.2 GB（1.3B × 4字节）
├── 演示视频 MP4 文件：约 6 GB    ← 不会加载进显存！
├── 演示图片 PNG 文件：约 2 GB    ← 不会加载进显存！
└── 配置/代码文件：约 0.8 GB

实际推理显存：~11 GB
├── 模型权重 FP16：约 2.6 GB（1.3B × 2字节）
├── 框架/CUDA 开销：约 1 GB
└── KV Cache + 激活值：约 7.4 GB（处理长音频/视频时较大）
```

**结论**：
1. MP4/PNG 演示文件是仓库的"附带内容"，推理时根本不加载
2. FP32→FP16 节省一半权重显存
3. KV Cache 和激活值可能比权重本身占用更多显存（处理长序列时）

## 九、推理加速技术

### 1. FlashAttention（最重要）

**问题根源**：标准 Attention 计算的显存复杂度是 O(n²)（n = 序列长度），长序列时显存爆炸。

**FlashAttention 原理**：

```
标准 Attention：
  Q、K、V → 计算完整 n×n 注意力矩阵 → 存入 HBM（显存）→ 读回 → Softmax → 读回 → × V
  显存访问次数：极多，带宽成为瓶颈

FlashAttention：
  将 Q、K、V 分块（tile）→ 每块在 SRAM（片上缓存）内完成计算 → 只写出结果
  显存访问次数：大幅减少（IO 复杂度从 O(n²) 降到 O(n)）
```

**效果**：
- 显存占用从 O(n²) 降为 O(n)，支持更长序列
- 推理速度提升 2~4 倍（减少了显存读写次数）
- 对模型精度无损（数学等价）

```bash
pip install flash-attn --no-build-isolation   # 安装，需要匹配 SM 版本
```

```python
# 使用 FlashAttention
model = AutoModel.from_pretrained("...", attn_implementation="flash_attention_2")
```

**注意**：flash_attn 是编译型包，需要 **SM 版本 ≥ sm80**（A100 及以上才能完整支持）。

### 2. torch.compile（PyTorch 2.0+）

将模型编译为优化后的 CUDA 代码，减少 Python 开销：

```python
model = torch.compile(model)   # 首次调用有编译开销，之后快 10~30%
```

### 3. vLLM / PagedAttention

专为 LLM 推理设计的服务框架：
- **PagedAttention**：KV Cache 按页管理，类似操作系统内存分页，大幅提升并发吞吐
- 支持连续批处理（Continuous Batching）
- 吞吐量比 naive 实现高 10~20 倍

```bash
pip install vllm
python -m vllm.entrypoints.openai.api_server --model /path/to/model
```

### 4. 量化（Quantization）

将 FP16 权重压缩为 INT8/INT4，减少显存占用：

| 方案 | 压缩比 | 精度损失 | 工具 |
|---|---|---|---|
| INT8 | 2× | 极小（<1%）| bitsandbytes |
| INT4 (GPTQ) | 4× | 小 | AutoGPTQ |
| INT4 (AWQ) | 4× | 最小 | AutoAWQ |
| GGUF (llama.cpp) | 可变 | 可控 | llama.cpp |

```python
# INT8 量化加载
model = AutoModel.from_pretrained("...", load_in_8bit=True, device_map="auto")

# INT4 量化加载
from transformers import BitsAndBytesConfig
bnb_config = BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_compute_dtype=torch.bfloat16)
model = AutoModel.from_pretrained("...", quantization_config=bnb_config)
```

### 5. 张量并行 / 流水线并行

当单卡显存不足时，将模型切分到多张 GPU：

```python
# HuggingFace 自动切分
model = AutoModel.from_pretrained("...", device_map="auto")
# device_map="auto" 会自动把层分配到各 GPU，VRAM 不足时 offload 到 CPU
```

### 6. CPU Offloading

将不常用的层放在 CPU 内存，需要时再搬到 GPU：

```python
model = AutoModel.from_pretrained("...", device_map="auto", offload_folder="offload_dir")
```
代价：推理速度降低（受 PCIe 带宽限制）。

### 加速技术效果对比

| 技术 | 显存节省 | 速度提升 | 精度 | 部署复杂度 |
|---|---|---|---|---|
| FP16 加载 | 50% | +20~50% | 几乎无损 | 简单 |
| BF16 加载 | 50% | +20~50% | 几乎无损 | 简单 |
| FlashAttention | 长序列大幅减少 | +2~4× | 无损 | 中等 |
| torch.compile | 几乎无 | +10~30% | 无损 | 简单 |
| INT8 量化 | 50% | ±0~10% | <1% 损失 | 中等 |
| INT4 量化 | 75% | ±0~20% | 1~3% 损失 | 复杂 |
| vLLM | — | 吞吐 10~20× | 无损 | 较复杂 |

## 十、A100 在 AI 部署中的特殊优势

A100（sm80）是目前 AI 部署最均衡的选择，原因：

1. **BF16 原生支持**：比 RTX 3090（sm86）更稳定的 BF16 Tensor Core 运算
2. **HBM2e 显存**：带宽 ~2 TB/s，比 GDDR6X（RTX 4090）高约 2 倍
3. **flash_attn 完整支持**：sm80 是 flash_attn v2 的最低要求
4. **NVLink**：多卡互联带宽高，适合多 GPU 并行
5. **sm 版本跨度大**：sm80 被几乎所有主流框架版本（torch 1.x~2.x）覆盖，兼容性最好
