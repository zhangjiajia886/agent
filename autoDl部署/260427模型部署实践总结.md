## 一、背景


## 二、遇到的困难与解决过程

### 困难 1：以为一个 Python 环境装得下三个模型

**现象**：写了一个 `setup_autodl.sh`，试图用同一个 `conda soul` 环境跑三个服务。

**真相**：读完三个服务的 `requirements.txt` 之后，发现版本冲突严重：

```
FlashHead  →  torch 2.7.1,  transformers 4.57.3,  gradio 5.50
Podcast    →  torch 2.7.1,  transformers 4.57.1
Singer     →  torch 2.2.0,  transformers 4.41.2,  numpy < 2.0  ← 完全不一样！
```

Singer 要求的 torch 2.2.0 与另外两个要求的 2.7.1 **根本不兼容**，强行装一起会互相覆盖。

**解决方案**：每个服务用独立的 conda 环境，彻底隔离：

```
/root/autodl-tmp/envs/
├── env-flashhead/   (torch 2.7.1+cu128, Python 3.10)
├── env-podcast/     (torch 2.7.1+cu128, Python 3.10)
└── env-singer/      (torch 2.2.0+cu121, Python 3.10)  ← 独立隔离
```

**原理**：conda 的 `--prefix` 参数可以把环境创建在任意目录，避免污染系统盘：
```bash
conda create --prefix /root/autodl-tmp/envs/env-flashhead python=3.10 -y
conda activate /root/autodl-tmp/envs/env-flashhead
```

### 困难 2：pip 安装 15 分钟没有任何输出，以为卡死了

**现象**：执行 `pip install torch -q` 后，15 分钟内终端没有任何输出，以为程序卡死。

**真相**：`-q` 参数（quiet）把所有下载进度都屏蔽了。torch 2.7.1 的 CUDA 依赖包合计超过 3 GB：

```
nvidia-cudnn-cu12      727 MB
nvidia-cublas-cu12     610 MB
nvidia-cusparse-cu12   292 MB
nvidia-cusolver-cu12   260 MB
nvidia-cufft-cu12      193 MB
nvidia-nccl-cu12       201 MB
triton                 149 MB
torch 主体             ~1 GB
─────────────────────────────
合计                  ~3.5 GB
```

**解决方案**：去掉 `-q`，可以看到下载进度条。长任务一律放入 `tmux`，避免 SSH 断线丢失。

**原理**：`pip install -q` 只是屏蔽输出，下载还在进行。判断"是否卡死"要看进程，而不是屏幕：
```bash
ps aux | grep pip   # 进程还在 = 没卡死
```

### 困难 3：numpy 下载反复超时

**现象**：torch 装好了，但接下来安装 `numpy-2.2.6` 时，从 `files.pythonhosted.org` 下载到 3.4 MB/16.8 MB 就反复超时：

```
WARNING: Connection timed out while downloading.
error: incomplete-download
× Download failed after 6 attempts (3.4 MB/16.8 MB)
note: This is an issue with network connectivity, not pip.
```

**原因**：AutoDL 服务器访问 `files.pythonhosted.org`（境外 PyPI 官方源）网速极慢。

**解决方案**：换清华大学镜像源，速度从几 kB/s 提升到 30~45 MB/s：

```bash
pip install -r requirements.txt \
  -i https://pypi.tuna.tsinghua.edu.cn/simple \
  --trusted-host pypi.tuna.tsinghua.edu.cn
```

**原理**：pip 支持 `-i` 参数指定包索引地址。清华镜像每隔几小时从官方同步一次，在中国境内访问速度极快。

### 困难 4：nvidia-nccl-cu12 版本冲突

**现象**：安装 FlashHead 的 `requirements.txt` 时报错：

```
nvidia-nccl-cu12==2.27.3 conflicts with nvidia-nccl-cu12==2.26.2 (installed by torch)
```

**原因**：torch 安装时已经依赖了 `nvidia-nccl-cu12==2.26.2`，而 FlashHead 的 `requirements.txt` 里写死了 `2.27.3`，pip 无法同时满足。

**解决方案**：安装前过滤掉这一行，torch 已带的版本完全够用：

```bash
grep -v "nvidia-nccl-cu12" requirements.txt > /tmp/req_filtered.txt
pip install -r /tmp/req_filtered.txt
```

**原理**：AI 研究仓库的 `requirements.txt` 通常是从开发者自己的环境直接导出的，版本锁定过于精确，实际运行时往往不需要精确到小版本。

---

### 困难 5：Singer 在 RTX 5090 上根本无法运行

**现象**：Singer 要求 `torch==2.2.0`，但 RTX 5090 是 Blackwell 架构（SM_120），torch 2.2.0 根本不支持。

```
RuntimeError: CUDA error: no kernel image is available for execution on the device
sm_120 not supported by torch 2.2.0 (max sm_90)
```

**原因**：每个 CUDA GPU 都有"计算能力"（Compute Capability，简称 SM）。不同版本的 PyTorch 支持不同范围的 SM：

```
SM版本  →  对应硬件           →  最低需要的 torch 版本
sm_70  →  V100（Volta）       →  torch 1.x
sm_80  →  A100（Ampere）      →  torch 1.x
sm_86  →  RTX 3090（Ampere）  →  torch 1.x
sm_90  →  H100（Hopper）      →  torch 2.0+
sm_120 →  RTX 5090（Blackwell）→  torch 2.6+  ← Singer 要的 2.2.0 不够！
```

**解决方案**：Singer 单独部署到 V100（sm_70）实例，完全隔离。RTX 5090 专跑 FlashHead + Podcast。

**教训**：选云 GPU 时，**要验证模型的 torch 版本与 GPU 的 SM 是否匹配**，这是最容易踩的坑。

---

### 困难 6：GitHub 克隆超时

**现象**：直接 `git clone https://github.com/...` 超时，没有响应。

**原因**：AutoDL 服务器访问 github.com 需要经过国际链路，经常超时。

**解决方案**：使用 `kkgithub.com` 镜像（完整镜像，只需替换域名）：

```bash
git clone https://kkgithub.com/Soul-AILab/SoulX-FlashHead ~/services/flashhead
```

**原理**：kkgithub.com 是 GitHub 的国内镜像站，内容实时同步，只是域名不同。

---

### 困难 7：服务找不到模型文件（相对路径问题）

**现象**：FlashHead 启动后报错：

```
FileNotFoundError: models/SoulX-FlashHead-1_3B/Model_Lite not found
```

**原因**：`gradio_app.py` 里写的是相对路径 `models/...`，从服务目录 `~/services/flashhead/` 启动时，会查找 `~/services/flashhead/models/`，而模型实际在 `~/models/`。

**解决方案**：用软链接"欺骗"程序，让相对路径生效：

```bash
ln -sf ~/models ~/services/flashhead/models
# 结果：~/services/flashhead/models → ~/models
```

**原理**：`ln -sf` 创建的是符号链接（软链接），类似 Windows 的快捷方式。程序顺着链接找，就能找到真实位置，无需改代码。

---

### 困难 8：AutoDL 端口限制——7860/7861/7862 外网不可达

**现象**：Gradio 默认启动在 7860 端口，但从公网完全无法访问。

**原因**：AutoDL 的"自定义服务"**只映射 6006 和 6008 两个端口**到公网，其他端口全部屏蔽：

```
AutoDL 端口映射规则：
内网 http://127.0.0.1:6006  →  公网 https://u982127-xxx.westd.seetacloud.com:8443
内网 http://127.0.0.1:6008  →  公网 https://uu982127-xxx.westd.seetacloud.com:8443
其他端口（7860/7861/7862）  →  ❌ 不可达
```

**解决方案**：
- FlashHead（Gradio）改到 6006：`GRADIO_SERVER_PORT=6006 python gradio_app.py`
- Podcast（FastAPI）改到 6008：`python run_api.py --port 6008`
- Singer（仅本地需要）：SSH 本地端口转发：`ssh -L 7862:localhost:7862 autodl -N`

### 困难 9：模型体积严重低估，磁盘爆满

**现象**：数据盘用到 100%（50G/50G），再也无法写入任何文件。

**原因**：最初按「参数量 × 2字节 = FP16 大小」估算，实际 HuggingFace 仓库包含更多内容：

```
估算 vs 实际：
                  估算        实际      误差
FlashHead-1_3B    2.6 GB  →  14 GB   （5.4倍！内含演示 MP4/PNG 文件）
wav2vec2          360 MB  →  1.1 GB
Podcast-1.7B      3.4 GB  →  5.4 GB
Singer            4 GB    →  5.3 GB
───────────────────────────────────
合计              ~11 GB  →  26 GB
```

**解决方案**：换 14 TB 大数据盘，并把 conda 环境也移到数据盘（`/root/autodl-tmp/envs/`）。

**教训**：HuggingFace 模型仓库实际包含的内容远比参数文件多，要实际下载后再规划磁盘。

---

### 困难 10：数据盘为空——忘记镜像不保存数据盘

**现象**：删掉旧实例，用镜像创建新实例，发现 `~/models` 是空的，所有环境消失。

**原因**：AutoDL 镜像只保存系统盘，**数据盘（`/root/autodl-tmp`）不包含在镜像内**：

```
AutoDL 镜像保存范围：
┌─────────────────────────────────────────────────────┐
│  系统盘（/）                                   ✅    │
│    /root/services/      服务代码               ✅    │
│    /root/miniconda3/    conda 基础             ✅    │
│    /root/*.sh           启动脚本               ✅    │
├─────────────────────────────────────────────────────┤
│  数据盘（/root/autodl-tmp）                    ❌    │
│    envs/                conda 环境             ❌ 丢失│
│    models/              模型文件               ❌ 丢失│
└─────────────────────────────────────────────────────┘
```

**解决方案**：
- **预防**：关机不删实例（数据盘永久保留）
- **发生了**：执行准备好的恢复脚本，约 60 分钟自动恢复

---

### 困难 11：flash_attn 无法安装，第一次推理极慢

**现象**：
1. `pip install flash_attn` 失败，RTX 5090（SM_120）不在支持列表
2. 第一次数字人生成耗时 **110 秒**，后续每帧仅 0.18 秒

**原因**：
1. flash_attn 预编译包最高支持 SM_90（H100），SM_120 暂无
2. 第一次推理触发 **CUDA JIT 编译**，PyTorch 现场为新 GPU 架构编译内核

```
FlashHead chunk-0：109.73s  ← CUDA kernel 编译（一次性开销）
FlashHead chunk-1：  0.18s  ← 编译完，正常速度
FlashHead chunk-2：  0.18s
...
```

**现状**：flash_attn 是 RTX 5090 的架构限制，目前无解。CUDA JIT 编译在第一次启动后内核会缓存，后续快很多。

## 三、学到的核心原理

### 原理 1：系统盘 vs 数据盘

```
┌──────────────────────────────────────────────────┐
│  系统盘（overlay 文件系统）                        │
│  • 基于镜像，可读写                                │
│  • 保存镜像时写入快照                              │
│  • 删除实例 → 内容丢失                             │
│  • 容量小（30~50 GB）                              │
│  路径：/  /root（非 autodl-tmp 部分）               │
└──────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────┐
│  数据盘（独立块存储）                               │
│  • 挂载到 /root/autodl-tmp                        │
│  • 关机/重启不丢失                                 │
│  • 删除实例 → 内容丢失                             │
│  • 容量大（50 GB ~ 14 TB）                         │
│  路径：/root/autodl-tmp/                           │
└──────────────────────────────────────────────────┘
```

**最佳实践**：所有大文件（模型、conda 环境）放数据盘，服务代码放系统盘（通过镜像保存）。

---

### 原理 2：GPU 计算能力（SM）与兼容性

每块 GPU 都有一个"计算能力"版本号（Compute Capability），决定了能运行哪些 CUDA 程序：

```
GPU 代际       架构名      SM 版本    经典卡型
─────────────────────────────────────────────
Tesla          Fermi       sm_20     —（太老）
Maxwell        Maxwell     sm_52     GTX 980
Pascal         Pascal      sm_60/61  GTX 1080 / P100
Volta          Volta       sm_70     V100
Turing         Turing      sm_75     RTX 2080
Ampere         Ampere      sm_80/86  A100 / RTX 3090
Hopper         Hopper      sm_90     H100
Ada            Ada         sm_89     RTX 4090
Blackwell      Blackwell   sm_120    RTX 5090  ← 最新，兼容性最差
```

**选卡原则**：如果模型代码要求 `torch <= 2.2`，就不能用 RTX 5090，要用 A100 或更早的卡。

---

### 原理 3：conda 环境隔离机制

conda 环境本质上就是一个独立的目录，里面有自己的 Python 解释器和所有包：

```
/root/autodl-tmp/envs/
└── env-flashhead/
    ├── bin/python3.10        ← 独立 Python
    ├── bin/pip               ← 独立 pip
    └── lib/python3.10/
        └── site-packages/    ← 独立包目录（约 8 GB）
            ├── torch/
            ├── gradio/
            └── ...
```

每个服务激活自己的环境，互不干扰：
```bash
conda activate /root/autodl-tmp/envs/env-flashhead
python run.py   # 用 env-flashhead 的包
```

---

### 原理 4：tmux 的作用

SSH 连接是临时的。断网后，普通命令会被 Kill：

```
本地 → SSH → 远程进程
当 SSH 断开 → 远程进程收到 SIGHUP → 进程被终止

解决方法：tmux（终端复用器）
本地 → SSH → tmux session → 命令进程
当 SSH 断开 → tmux session 仍然存在 → 命令继续运行
重新 SSH → tmux attach → 可以继续查看
```

长任务必须用 tmux：
```bash
tmux new-session -d -s download         # 建 session
tmux send-keys -t download "命令" C-m   # 在 session 里执行
tmux attach -t download                 # 随时查看
```

---

### 原理 5：镜像加速原理

| 资源 | 官方源 | 推荐加速方式 | 速度对比 |
|---|---|---|---|
| Python 包（pip）| files.pythonhosted.org | 清华镜像 `-i https://pypi.tuna.tsinghua.edu.cn/simple` | 几 kB/s → 30+ MB/s |
| GitHub 代码 | github.com | kkgithub.com（替换域名）| 超时 → 正常 |
| HuggingFace 模型 | huggingface.co | hf-mirror.com（`HF_ENDPOINT`）| 几 kB/s → 5+ MB/s |
| torch CUDA 包 | download.pytorch.org | 原站可直连（AutoDL 走海外带宽）| 4+ MB/s ✅ |

## 四、专有名词速查

| 名词 | 通俗解释 |
|---|---|
| **AutoDL** | GPU 云服务器租用平台，按小时计费 |
| **conda 环境** | 独立的 Python 沙箱，不同项目互不干扰 |
| **SM（Compute Capability）** | GPU 的"语言版本号"，越新的 GPU SM 越高 |
| **CUDA** | NVIDIA GPU 的编程框架，不同版本支持不同 SM |
| **torch** | PyTorch，最主流的 AI 训练/推理框架 |
| **flash_attn** | Transformer 注意力机制的加速库，需要特定 GPU 支持 |
| **CUDA JIT 编译** | GPU 第一次运行新代码时在线编译，耗时一次，之后快 |
| **requirements.txt** | Python 项目的依赖清单，记录所有包的版本 |
| **HuggingFace** | AI 模型托管平台，类似 GitHub 但专门存 AI 模型 |
| **safetensors** | 模型权重文件格式，比 .bin 更安全更快 |
| **Gradio** | 快速搭建 AI 演示界面的 Python 框架 |
| **tmux** | 终端复用器，SSH 断线后任务仍然运行 |
| **软链接（symlink）** | 指向另一个路径的"快捷方式"，`ln -sf 真实路径 链接名` |
| **overlay 文件系统** | 云服务器系统盘的技术，分层存储，删实例后恢复初始 |
| **端口映射** | 把云服务器内网端口暴露到公网，AutoDL 只开放 6006/6008 |
| **SSH 隧道** | 把远程端口映射到本地，`ssh -L 本地端口:远程主机:远程端口` |

## 五、给下次部署的自己

### 部署前必检清单

```
[ ] 确认模型的 torch 版本与 GPU SM 版本兼容
[ ] 预留足够数据盘空间（实际 = 估算 × 3）
[ ] 确认哪些端口可以对外访问
[ ] GitHub 仓库是否公开可克隆
[ ] 准备好镜像源配置
```

### 效率最高的顺序

```
1. 并行下载所有模型（最耗时，先启动）
2. 安装 conda 环境（等模型下载同时进行）
3. 建软链接 + 调整端口
4. 启动服务 + 验证
5. 保存镜像快照（防止重装）
```

### 一句话总结每个坑

```
依赖冲突  → 每个服务独立 conda 环境
pip 超时  → 换清华镜像
无输出    → 去掉 -q，加 tee 写日志
SSH 断线  → 用 tmux
找不到模型 → 用软链接
磁盘估算  → 乘以 3 倍再规划
数据丢了  → 关机但不删实例
端口不通  → AutoDL 只有 6006/6008 可用
GPU 不兼容 → 查 torch 支持的 SM 版本范围
```
