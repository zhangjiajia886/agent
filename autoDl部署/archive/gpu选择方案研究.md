# GPU 选择方案研究

> 场景：同时运行 FlashHead（数字人视频）+ Podcast（声音克隆）+ Singer（歌声合成）  
> 目标：不差钱，追求最佳体验

---

## 一、四种存储的作用（彻底搞清楚）

```
用户请求
   ↓
CPU（大脑）
   ↓ 指令
GPU（算力核心）← 显存 VRAM（GPU 的工作台）
   ↑
内存 RAM（CPU 的工作台）
   ↑
数据盘（仓库，持久化）← 模型文件长期存在这里
系统盘（车间，临时）← 代码、环境
```

### 显存（VRAM）— GPU 的工作台

| 特性 | 说明 |
|---|---|
| **作用** | AI 推理时模型权重必须全部加载到显存才能运算 |
| **速度** | 最快，GPU 直接访问，带宽 1-3 TB/s |
| **容量** | 16-80 GB（贵，固定在 GPU 上） |
| **断电** | 丢失（每次启动重新从数据盘加载） |
| **对推理速度影响** | **决定性**——显存不够模型跑不了 |

**本项目实测占用：**
```
FlashHead 推理峰值：~11 GB
Podcast 常驻：        ~5.3 GB
Singer 推理峰值：     ~6 GB
三服务同时运行峰值：  ~22-24 GB
```

### 内存（RAM）— CPU 的工作台

| 特性 | 说明 |
|---|---|
| **作用** | 数据预处理、图像解码、音频处理、Python 运行时 |
| **速度** | 快，~100 GB/s |
| **容量** | 32-2000 GB（比显存便宜） |
| **断电** | 丢失 |
| **对推理速度影响** | 间接——预处理瓶颈才会影响 GPU 利用率 |

**本项目实测占用：**
```
三个 Python 进程合计：~20-30 GB
操作系统底层：         ~5 GB
建议配置：             64 GB 以上
```

### 系统盘（System Disk）— 车间

| 特性 | 说明 |
|---|---|
| **作用** | 操作系统、conda 环境、服务代码、pip 包 |
| **速度** | 快 SSD |
| **容量** | 30-50 GB 够用 |
| **断电** | ⚠️ AutoDL 实例关机后**系统盘恢复到镜像状态**（代码丢失） |
| **持久化** | 只有保存镜像快照才能保留修改 |

**本项目实测占用：**
```
系统盘（overlay 30GB）：仅用 1.8 GB（6%）
→ 系统盘完全不是瓶颈
```

### 数据盘（Data Disk）— 仓库

| 特性 | 说明 |
|---|---|
| **作用** | 模型权重（长期存储）、conda 环境（存这里不怕系统盘重置）、生成的视频/音频 |
| **速度** | NVMe SSD，3-7 GB/s |
| **容量** | 50-1000 GB（按需扩容） |
| **断电** | ✅ **持久化，关机不丢** |
| **对推理速度影响** | 仅影响启动时模型加载速度（加载完进显存就没关系了） |

**本项目实测占用：**
```
模型文件：  28 GB（FlashHead 14G + Podcast 5.4G + Singer 5.3G + 预处理 2.5G + wav2vec 1.1G）
conda 环境：23 GB（三个独立环境）
生成文件：  动态增长（视频 ~2MB/个，音频 ~1MB/个）
合计：      ~51 GB（150GB 数据盘占用 34%）
```

## 二、推理速度的决定因素（按重要性排序）

```
1. 显存大小       → 模型能否加载（必要条件）
2. GPU 算力       → TFLOPS 越高推理越快（核心）
3. flash_attn     → Attention 加速，差 5-10 倍
4. 数据盘速度     → 只影响首次加载，NVMe 都够用
5. 内存大小       → 预处理够快就行，64GB 足够
6. 系统盘大小     → 无关推理速度
```

---

## 三、关键问题：数据盘 51GB，显存 40GB 够吗？

> **数据盘大小 ≠ 显存需求。两者完全不同的概念。**

### 为什么数据盘比显存大？

模型文件在数据盘上的大小 ≠ 推理时占用的显存大小，原因：

| 原因 | 说明 | 举例 |
|---|---|---|
| **精度不同** | 磁盘存 FP32（4字节/参数），推理用 FP16（2字节/参数） | 磁盘 14GB → 显存 7GB |
| **含非权重文件** | 仓库里有演示视频、README、配置文件等 | FlashHead 仓库含大量 MP4/PNG |
| **conda 环境** | Python 包、库文件，和推理无关 | env-flashhead 8.6GB 不进显存 |
| **模型分层加载** | 推理时只加载当前需要的层 | 显存可比磁盘更小 |

### 实测数据对比（RTX 5090 实测）

| 服务 | 数据盘占用 | 实测显存占用 | 差异原因 |
|---|---|---|---|
| FlashHead | 14 GB | **~11 GB** | FP16推理 + 无演示文件 |
| Podcast | 5.4 GB | **~5.3 GB** | 接近（纯模型仓库） |
| Singer | 5.3 GB | **~6 GB** | 略高（含多子模型） |
| conda envs | 23 GB | **0 GB** | Python包不进显存 |
| Singer-Preprocess | 2.5 GB | **按需加载** | 预处理时部分进显存 |
| **磁盘合计** | **51 GB** | **峰值 ~24 GB** | 差了一倍 |

### 结论：40GB 显存是否足够？

```
三服务同时推理峰值显存：
  FlashHead 推理：11 GB
  Podcast 常驻：   5 GB
  Singer 推理：    6 GB
  CUDA 框架开销：  2 GB
  ─────────────────────
  峰值合计：      ~24 GB

A100 40GB 占用率：24 / 40 = 60% ✅

注意：三个服务同时推理是最极端情况
通常 FlashHead 和 Singer 不会同时触发推理
实际日常占用更接近 15-18 GB（45%）
```

**51GB 放不下是数据盘的问题（存储），不是显存的问题（计算）。**  
扩容数据盘到 150GB 已解决存储问题，与显存无关。

| GPU | 显存 | FP16 算力 | Singer 兼容 | flash_attn | AutoDL 价格 |

### 🥇 方案一：H100 SXM5 80GB（终极方案）

```
GPU：    H100 SXM5 80GB × 1
显存：   80 GB（三服务峰值 24GB，占用 30%）

数据盘： 200 GB
系统盘： 30 GB（够用）
```

**优势：**
- 三服务同时推理显存占用 < 30%，极为宽裕

- Singer torch 2.2.0 完全兼容（sm_90 支持）
- NVLink 带宽超高，多模型并发无瓶颈

**适合场景：** 生产环境、多用户并发

## 六、当前实例（RTX 5090）问题总结

| 问题 | 根因 | 解决方案 |
|---|---|---|
| Singer 跑不了 | torch 2.2.0 不支持 sm_120 | 换 A100/H100 |
